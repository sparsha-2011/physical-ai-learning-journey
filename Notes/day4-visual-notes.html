01 — Edit Target

The edit target is one of the most important — and most misunderstood — concepts in USD. It answers the question: when I call Set() or DefinePrim(), which file on disk actually receives the change?

Real-world analogy

Imagine you're in a band rehearsal with three microphones: one for the lead guitarist, one for the drummer, one for the vocalist. Whoever the sound engineer points the recording arm at is the "edit target" — only THEIR mic gets recorded, even if everyone is playing simultaneously. You can switch the recording arm to a different mic at any time, but only one mic is recording at once.

A stage is composed from many layers. When you make any edit — Set() an attribute, define a prim, author metadata — that edit goes to exactly ONE layer: the current edit target. By default this is the root layer. But you can switch it to any layer in the stack at any time.

Edit target — only one layer receives your edits at a time
Most common USD pipeline bug

Writing to the wrong edit target is the single most common source of "my opinion isn't taking effect" bugs. You think you're editing anim.usda but the edit target is still pointing at the root layer — so your change goes to the wrong file, silently, with no error. Always verify your edit target before authoring data in multi-layer pipelines.

PythonReading and switching the edit target
from pxr import Usd, UsdGeom, Sdf

stage = Usd.Stage.Open("shot.usda")

# ── CHECK THE CURRENT EDIT TARGET ───────────────────────────────────
current = stage.GetEditTarget()
print(current.GetLayer().identifier)
# Prints the file path of the layer currently receiving edits
# Default = root layer (shot.usda itself)

# ── SWITCH EDIT TARGET TO A SUBLAYER ────────────────────────────────
layer_stack = stage.GetLayerStack()
# GetLayerStack() returns all layers in strength order — root is first

# Find the anim layer specifically
anim_layer = Sdf.Layer.FindOrOpen("./anim.usda")
stage.SetEditTarget(anim_layer)
# Now ALL edits go to anim.usda
# shot.usda (root) is NOT modified

# ── AUTHOR DATA TO THE TARGET LAYER ─────────────────────────────────
robot = stage.GetPrimAtPath("/World/Robot")
robot.GetAttribute("xformOp:translate").Set((5, 0, 0))
# This opinion goes into anim.usda — not the root layer
# In USDA: anim.usda now contains  over "Robot" { double3 xformOp:translate = (5,0,0) }

# ── USE A CONTEXT MANAGER (cleaner, safer) ───────────────────────────
with stage.GetEditTargetContext(anim_layer):
    # All edits inside this block go to anim_layer
    robot.GetAttribute("xformOp:translate").Set((10, 0, 0))
# After the 'with' block, edit target automatically reverts to what it was before
# This is the safest way to temporarily redirect edits

# ── SWITCH BACK TO ROOT ──────────────────────────────────────────────
stage.SetEditTarget(stage.GetRootLayer())
# Resets to default — edits go to the root layer again

Edit target with references — authoring into referenced assets

You can also set the edit target to a layer that's inside a referenced asset. This lets you edit the referenced asset's file directly from the referencing stage — useful in pipeline tools but dangerous if misused.

PythonEditing inside a referenced asset's layer
# stage has /World/Chair which references chair.usda
chair_prim = stage.GetPrimAtPath("/World/Chair")

# Get the layer stack that contributes to this prim
# PrimStack = all prim specs from all layers in strength order
prim_stack = chair_prim.GetPrimStack()
for spec in prim_stack:
    print(spec.layer.identifier)
# Prints ALL layers that have opinions about /World/Chair
# Includes: shot.usda (root), chair.usda (referenced), chair_geo.usda (inside reference)

# Set edit target to chair.usda directly
chair_layer = Sdf.Layer.FindOrOpen("./chair.usda")
stage.SetEditTarget(chair_layer)

# Now edits write to chair.usda — modifying the original asset
# Be careful: this changes the asset for ALL scenes that reference it

02 — The Session Layer

The session layer is a special in-memory layer that exists above all other layers — it is the strongest layer in the entire stage. It is never saved to disk. It's designed for temporary, interactive overrides that should not persist.

Real-world analogy

The session layer is like a sticky note you put on top of a printed document. The note says "ignore this section" or "change this number to X." The sticky note is the strongest voice — it covers what's underneath. When you finish the meeting and throw away the note, the original document is completely unchanged. The session layer works exactly like this — temporary, non-destructive, disposable.

Session layer — always the strongest, never saved to disk

The session layer is created automatically when you open a stage. It is always the edit target for interactive tools like usdview — when you change a variant in usdview's metadata panel, that change goes to the session layer, not to any file on disk.

PythonWorking with the session layer
from pxr import Usd, UsdGeom

stage = Usd.Stage.Open("scene.usda")

# ── GET THE SESSION LAYER ────────────────────────────────────────────
session = stage.GetSessionLayer()
print(session.identifier)
# Prints something like: anon:0x... (anonymous — no file path)
# The session layer is anonymous — it has no file on disk

# ── SET THE SESSION LAYER AS EDIT TARGET ─────────────────────────────
stage.SetEditTarget(session)
# Now all edits are temporary — nothing touches any file on disk

# ── AUTHOR A TEMPORARY OVERRIDE ─────────────────────────────────────
prim = stage.GetPrimAtPath("/World/Robot")
prim.GetAttribute("xformOp:translate").Set((99, 0, 0))
# The robot appears at (99, 0, 0) in the viewport
# But NO file on disk was changed

# ── CHECK WHAT'S IN THE SESSION LAYER ───────────────────────────────
print(session.ExportToString())
# Shows the in-memory content — the temporary overrides

# ── CLEAR THE SESSION LAYER ─────────────────────────────────────────
session.Clear()
# Wipes all session overrides — robot returns to its original position
# No file was touched — completely clean

# ── COMMON USE CASE: switch a variant temporarily ────────────────────
stage.SetEditTarget(session)
vset = prim.GetVariantSets().GetVariantSet("color")
vset.SetVariantSelection("blue")
# Variant selection goes to session layer — nothing on disk changes
# This is exactly what usdview does when you change a variant in the UI
Exam tip — usdview and the session layer

When you change a variant or visibility in usdview's metadata panel, it writes to the session layer. This is why those changes disappear when you close usdview — the session layer is never saved. If you want to persist a variant selection, you need to write it to an actual layer file using SetEditTarget() pointing at a real layer.

03 — Sparse Overrides: USD's Superpower

Sparse overrides are the core of what makes USD powerful for production. The idea is simple: you only author the data that's different from the composed result. You don't duplicate anything. You just say "change this one property" — and everything else stays exactly as it was.

Real-world analogy

A sparse override is like a Git diff. You don't send someone a new copy of the entire codebase when you fix a bug — you send a patch that says "on line 42, change X to Y." The rest of the file is untouched. USD sparse overrides work the same way: author only what changed, leave everything else to be composed from the layers below.

This is sometimes called non-destructive editing — the original asset is never modified. You layer your changes on top using over prims in a stronger layer.

USDASparse override — change ONE colour on an entire complex asset
# robot.usda — the original asset (complex: 500 prims, thousands of properties)
# You want to change ONLY the left arm's colour to red for one specific shot
# You do NOT copy or modify robot.usda

# shot_override.usda — a tiny sparse override layer
#usda 1.0
over "World" {
    over "Robot" {
        over "Arm_L" {
            over "Mesh" {
                # The ONLY opinion in this entire file
                color3f[] primvars:displayColor = [(1, 0, 0)]
            }
        }
    }
}
# This file is tiny — just one property changed
# The entire robot (499 other prims) is completely untouched
# Sublayer this above robot.usda and you're done
PythonCreating sparse overrides programmatically
from pxr import Usd, UsdGeom, Sdf

stage = Usd.Stage.Open("scene.usda")

# ── OVERRIDE A PRIM USING OverridePrim() ─────────────────────────────
# OverridePrim() = create an 'over' prim spec — sparse override
override = stage.OverridePrim("/World/Robot/Arm_L/Mesh")
# Creates the minimum 'over' spec at that path
# Does NOT create any parent prims as 'def' — keeps them as 'over' too
# Result in USDA: over "World" { over "Robot" { over "Arm_L" { over "Mesh" { } } } }

# ── SET THE ONE THING YOU WANT TO CHANGE ────────────────────────────
mesh = UsdGeom.Mesh(override)
mesh.GetDisplayColorAttr().Set([(1.0, 0.0, 0.0)])
# Only this one attribute is authored — nothing else in the file changes

# ── WHAT THIS PRODUCES IN USDA ───────────────────────────────────────
# over "World" {
#     over "Robot" {
#         over "Arm_L" {
#             over "Mesh" {
#                 color3f[] primvars:displayColor = [(1, 0, 0)]
#             }
#         }
#     }
# }

stage.Save()
# The entire override is this tiny — just one property
💡
Why this matters for the exam: Sparse overrides are the mechanism behind non-destructive editing, multi-user workflows, shot-specific variations, and debugging. Every time you see an over prim in a USDA file, that's a sparse override — someone changing one thing without touching everything else. Understanding this pattern is essential for the Pipeline Development and Debugging domains.

Attribute blocks — blocking a value from below

A special type of sparse override: you can block an attribute to prevent any weaker layer from providing a value for it. The result is that the attribute appears to have no value at all.

PythonBlocking an attribute
# Scenario: a referenced asset has translate = (5, 0, 0)
# You want to make it appear as if NO translate was authored
# (not zero — completely absent)

attr = prim.GetAttribute("xformOp:translate")
attr.Block()
# In USDA this produces:  double3 xformOp:translate = None
# 'None' as a value = block — prevents ALL weaker layers from contributing
# Even if a weaker layer says translate = (10, 0, 0), it is silently ignored
# The attribute now has no value — stronger layers can still override the block

04 — Flatten: Baking Composition

Flattening takes a fully composed stage — with all its sublayers, references, payloads, variants, inherits — and bakes everything into a single, self-contained layer. The output has no composition arcs. All values are resolved and written out as plain data.

Real-world analogy

Flattening is like merging all your Git branches into one commit and squashing the history. The final state is the same, but all the individual contributions (who did what, in which file) are gone. The result is simpler and more portable, but you lose the modular structure. It's a one-way operation — you can't un-flatten.

When to flatten

  • Delivering an asset to an external studio (exam topic 1.9)
  • Archiving a specific version of a shot
  • Debugging — see all resolved values at once
  • Publishing a final asset with no dependencies
  • Creating a self-contained .usdz package

When NOT to flatten

  • Ongoing production work — you lose modularity
  • Assets that will continue to be updated
  • When you need to preserve the layer structure for collaboration
  • Large scenes — flattening can be memory and compute intensive
  • When you still need to swap variants or make overrides
Python + CLIThree ways to flatten
from pxr import Usd

stage = Usd.Stage.Open("complex_scene.usda")

# ── METHOD 1: Flatten to an Sdf.Layer object (in memory) ─────────────
flat_layer = stage.Flatten()
# Returns an Sdf.Layer with all composition baked in
# No references, no sublayers, no variants — just pure resolved data

# ── METHOD 2: Export the flat layer to a file ────────────────────────
flat_layer.Export("flat_output.usda")
# Writes the baked result to disk as a single standalone file
# Opening flat_output.usda will show all data with no composition arcs

# ── METHOD 3: Export directly from stage ─────────────────────────────
stage.Export("flat_output.usdc")
# Shortcut — flattens AND exports to the specified format in one call
# The file extension determines the format (.usda = text, .usdc = binary)

────────────────────────────────────────────

# ── COMMAND LINE: usdcat with --flatten ──────────────────────────────
# Run in terminal (not Python):
# usdcat --flatten complex_scene.usda -o flat_output.usda
#
# Also useful for format conversion:
# usdcat scene.usda -o scene.usdc   (convert text to binary)
# usdcat scene.usdc -o scene.usda   (convert binary to text — for debugging)

What flattening does to composition arcs

Exam trap — variants after flattening

When you flatten, only the currently selected variant gets baked. All other variant choices are permanently lost. If a car asset has color variants red/blue/green and the current selection is "red" — the flat output only contains red. The blue and green geometry is gone. This is why you never flatten an asset that will continue to be used in production.

05 — Encapsulated vs Unencapsulated Prims

This section deepens the encapsulation concept from Day 3. The official USD definition of encapsulation is specifically about where prims live relative to the reference target.

📖
Official definition: A prim is encapsulated when it is an ancestor of the referenced target prim — meaning it lives inside the referenced hierarchy. An unencapsulated prim lives outside the referenced prim's hierarchy in the same file.
USDAEncapsulated vs unencapsulated — the difference
# ── FULLY ENCAPSULATED — everything inside the referenced hierarchy ──
# encapsulated.usda
def Xform "World"          # ← the defaultPrim / referenced target
{
    def Mesh "Cube"
    {
        rel material:binding = </World/Looks/RedMat>
    }
    def Scope "Looks"       # ← Looks is INSIDE /World = encapsulated
    {
        def Material "RedMat" { }
    }
}
# When scene.usda references encapsulated.usda:
# /World/Cube AND /World/Looks/RedMat BOTH come in together
# The material binding works because Looks is inside the reference boundary

────────────────────────────────────────────

# ── UNENCAPSULATED — something important lives OUTSIDE the hierarchy ──
# unencapsulated.usda
def Xform "World"          # ← the referenced target
{
    def Mesh "Cube"
    {
        rel material:binding = </Looks/RedMat>
        # ← binding points OUTSIDE /World to /Looks
    }
}

def Scope "Looks"           # ← Looks is OUTSIDE /World = UNENCAPSULATED
{
    def Material "RedMat" { }
}
# When scene.usda references unencapsulated.usda targeting /World:
# ONLY /World and its children come in
# /Looks does NOT come in — it's outside the reference boundary
# The material binding now BREAKS — /Looks/RedMat doesn't exist in the scene
Production best practice — always encapsulate dependencies

Assets should always be fully encapsulated — every prim that the asset needs (materials, skeletons, look files) should live inside the asset's root prim hierarchy. If materials live at the root level outside the main prim, they won't come in when the asset is referenced. This is a common source of broken material bindings in pipeline work.

✅ Encapsulated (correct)

Materials, skeletons, and all dependencies live inside the asset's root prim. When referenced, everything comes in together. No broken references.

/Asset/Geometry
/Asset/Looks ← inside

❌ Unencapsulated (dangerous)

Materials or other dependencies live outside the asset's root prim. When referenced, only the root prim and its children come in. Dependencies are left behind.

/Asset/Geometry
/Looks ← outside, broken!

06 — Asset Structure Principles

This directly feeds into the Pipeline Development exam domain (14%). A well-structured USD asset enables collaboration, scalability, and reuse. The official NVIDIA guidelines give clear principles.

The five core principles

1
Simplicity — avoid complexity that confuses users
Use clear, intentional composition arcs. Avoid deeply nested composition that nobody can debug. Choose naming conventions that communicate intent — capitalize "public" prims, underscore "internal" ones. Don't use every composition arc just because you can.
2
Modularity — assets should be self-contained and reusable
Each asset should work as a standalone unit. Dependencies should be encapsulated inside the asset's hierarchy. A chair asset should contain its own materials — it should not rely on a material living elsewhere in someone else's scene. This enables reuse across multiple shots and projects.
3
Clear entry points and stable interfaces
Every asset should have a well-defined defaultPrim — the entry point for referencing. Variant sets and primvars that are meant to be controlled from outside should be "lofted" to the asset's root prim so they're accessible without loading payloads. Think of this as the asset's public API.
4
Scalability — support parallel workstreams
Split complex assets across discipline layers (geometry, rigging, shading, animation) so multiple artists can work simultaneously without file conflicts. The layer stack structure enables this — each discipline owns their layer, the root layer just lists them in the right order.
5
Anchored paths — use relative paths, not absolute
Always use relative file paths (./chair.usda) not absolute paths (/home/studio/project/chair.usda). Absolute paths break the moment the project moves to a different machine or directory. Relative paths keep the asset self-contained and portable.

The reference/payload pattern (exam topic 1.9)

A best-practice asset structure splits the asset into two layers — an interface layer (lightweight) and a payload layer (heavy geometry):

USDAThe reference/payload pattern — interface above, heavy data below
# chair.usda — the INTERFACE layer (small, fast to open)
#usda 1.0
(
    defaultPrim = "Chair"
)
def Xform "Chair" (
    # Heavy geometry is behind a payload — not loaded by default
    prepend payload = @./chair_geo.usda@

    # Variant sets are LOFTED here — accessible even without loading the payload
    prepend variantSets = "color"
    variants = { string color = "natural" }
)
{
    # Extent hint — lets tools know the bounding box without loading geometry
    float3[] extentsHint = [(-0.5,-0.5,-0.5), (0.5,1.0,0.5)]

    # The variant data (lofted here — works without loading payload)
    variantSet "color" = {
        "natural" { }
        "painted" { }
    }
}

────────────────────────────────────────────

# chair_geo.usda — the PAYLOAD layer (heavy — only loaded when needed)
def Xform "Chair"
{
    def Mesh "seat" { /* 50,000 vertices */ }
    def Mesh "legs" { /* 10,000 vertices */ }
    def Scope "Looks"
    {
        def Material "WoodMat" { }
    }
}
Why lofting variants matters

If variant sets are defined inside the payload, tools can't see them without loading the heavy geometry. Lofting means copying the variant set declaration up to the interface layer (above the payload). Now a scene can switch colour variants without ever loading the geometry — saves memory and load time. This is exam topic 1.9 territory — preparing assets for efficient pipeline use.

07 — Change Processing & SdfChangeBlock

Every time you edit a layer, USD's stage automatically re-composes affected prims. This is called change processing. It keeps the stage accurate — but it has a cost. If you make thousands of edits in a loop, each one triggers a re-composition. This can be very slow.

Real-world analogy

Change processing is like a live preview in a design tool. Every brush stroke instantly updates the screen. That's great for 1–2 edits. But if you're running a script that makes 10,000 changes, you don't want the screen to re-render after each individual stroke — you want it to wait until all the strokes are done, then update once. SdfChangeBlock is the "pause live preview" button.

PythonSdfChangeBlock — batch edits for performance
from pxr import Usd, UsdGeom, Sdf

stage = Usd.Stage.Open("city.usda")

# ── WITHOUT SdfChangeBlock — slow ────────────────────────────────────
for i in range(1000):
    prim = stage.GetPrimAtPath(f"/World/Building_{i}")
    prim.GetAttribute("height").Set(10.0)
    # After each Set(), USD re-composes the stage
    # 1000 iterations = 1000 re-compositions = VERY SLOW


# ── WITH SdfChangeBlock — fast ───────────────────────────────────────
with Sdf.ChangeBlock():
    for i in range(1000):
        prim = stage.GetPrimAtPath(f"/World/Building_{i}")
        prim.GetAttribute("height").Set(10.0)
        # No re-composition happens inside the 'with' block
        # All 1000 edits are queued up
# When the 'with' block exits → ONE re-composition for all 1000 edits
# Result: same final state, much faster execution


# ── WHEN TO USE SdfChangeBlock ───────────────────────────────────────
# Use it whenever you're making many edits in a loop
# Don't use it when you NEED the stage to be current inside the loop
# (e.g. if you're reading a value you just wrote and need it to be composed)
Exam topic 6.1 — when SdfChangeBlock alleviates performance bottlenecks

The exam may ask you to identify when SdfChangeBlock helps. The answer: any time you're making many sequential edits in a batch operation — updating transforms on thousands of prims, authoring animation curves, generating procedural geometry. The key is that change processing happens once at the end instead of after every single edit.

08 — Full Composition Mental Model

You've now covered all of composition across Days 2, 3, and 4. Here is the complete mental model that ties everything together — this is how to think about any composition question on the exam.

The complete layer stack — strength ordering including session layer

STRONGEST
Session Layer (in-memory, never saved)
interactive tool overrides
2nd
Root Layer sublayers (strongest first)
shot overrides, anim layer
3rd
Root Layer sublayers (weaker)
layout, lighting, model
4th
Referenced assets (via arcs — LIVERPS order)
I → V → R → P → S
WEAKEST
Schema fallback values
defined by prim type

The 5-question checklist for any composition exam question

1
Find every opinion about the property in question
Look in every layer — root layer, sublayers, referenced files, variant contexts. List them all before deciding anything.
2
Identify which LIVERPS category each opinion comes from
Is it a local opinion in a sublayer? A reference? A variant? A payload? An inherits? A specializes? Label each one.
3
Apply LIVERPS — the strongest category wins
L beats I beats V beats R beats P beats S. If two opinions are both Local, the one in the stronger sublayer wins (first listed = stronger).
4
Check for encapsulation boundaries
Are you trying to override something inside a reference from outside? Local opinions on the referencing prim work. Reaching into the reference's internal sublayer ordering doesn't.
5
If no arc wins — use the schema fallback
If no layer has any opinion at all, USD uses the fallback defined by the prim's schema type. This is why a Sphere has a radius of 1.0 even if you never set it.

Common "why doesn't my opinion work?" checklist

Day 4 key takeaways — commit these to memory

1The edit target determines which layer receives your edits. Default = root layer. Always check it before authoring in multi-layer pipelines — wrong edit target is the most common silent bug.
2Use with stage.GetEditTargetContext(layer): as the safest way to temporarily redirect edits — it automatically reverts the edit target when the block exits.
3The session layer is the strongest layer in the stage and is never saved to disk. usdview writes all interactive changes (variant selections, visibility) to the session layer.
4Sparse overrides are the core of non-destructive editing — use over prims and OverridePrim() to change only what needs to change. Never duplicate an entire asset to change one property.
5Flatten bakes all composition into one layer — use it for delivery to external parties, archiving, or debugging. Never flatten assets that will continue to be edited in production.
6Flattening discards all non-selected variant choices permanently. Only the currently active variant's data survives in the flat output.
7Encapsulated prims live inside the referenced hierarchy and come in when referenced. Unencapsulated prims live outside and are left behind — causing broken material bindings.
8Good asset structure: simple, modular, encapsulated, clear defaultPrim, lofted variant sets above payloads, relative file paths. These principles feed directly into the Pipeline Development exam domain.
9Change processing re-composes the stage after every edit. Wrap batch edits in Sdf.ChangeBlock() to defer re-composition until all edits are done — essential for performance with thousands of edits.
10Full composition strength order: Session Layer → Local sublayers (strongest first) → LIVERPS arcs (I → V → R → P → S) → Schema fallbacks. Session is always strongest, schema fallback is always weakest.
