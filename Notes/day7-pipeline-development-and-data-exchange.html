01 — USDC vs USDA Trade-offs

Choosing the right file format is a pipeline decision that affects load performance, debuggability, storage size, and collaboration workflow. This is exam topic 4.3 and 7.3 — you need to know when to use each.

Real-world analogy

USDA is like a Word document — anyone can open and read it, easy to edit, but takes more storage. USDC is like a compiled program — much faster to execute (load), but you can't just "read" it in a text editor. USDZ is like a zip attachment — portable and self-contained, great for sending to someone else, but not for ongoing editing.

Property .usda (ASCII) .usdc (Crate Binary) .usdz (Zip Archive)
Human readable?✅ Yes — open in any text editor❌ No — binary format❌ No — zip of binary/text
Load speedSlow — parsed as textVery fast — memory-mapped binaryFast — reads from zip
File sizeLarge — text is verboseSmall — compact binaryCompressed varies
Good for debugging?✅ Best — diff, inspect, edit❌ Needs usdcat to convert first❌ Extract first
Good for production?❌ Too slow for large data✅ Best for geometry/animationDelivery only
Can reference other files?✅ Yes✅ YesSelf-contained (read-only)
Git-diffable?✅ Yes — great for version control❌ Binary diff only❌ Binary
Best use caseInterface layers, debugging, small orchestration filesHeavy geometry, animation, shading networksFinal delivery, AR/web, external sharing
🏭
Production best practice from NVIDIA docs: Use .usdc (or .usd which defaults to usdc) for layers with heavy data. Reserve .usda for lightweight interface layers that mostly reference or sublayer other files, and for debugging. "Prefer crate files for big data, keep text to lightweight aggregators."
CLI + PythonConverting between formats
# ── COMMAND LINE CONVERSION ──────────────────────────────────────────
# Convert text to binary (for production performance)
usdcat scene.usda -o scene.usdc

# Convert binary to text (for debugging — make it readable)
usdcat scene.usdc -o scene_readable.usda

# Flatten AND convert (for delivery)
usdcat --flatten scene.usda -o flat_delivery.usdc

────────────────────────────────────────────

# ── PYTHON ───────────────────────────────────────────────────────────
from pxr import Usd

stage = Usd.Stage.Open("scene.usda")

# Export to binary — file extension determines format
stage.Export("scene.usdc")    # → binary
stage.Export("scene.usda")    # → text
stage.Export("delivery.usdz") # → packaged archive

# Convert a single layer (not the whole composed stage)
layer = stage.GetRootLayer()
layer.Export("root_only.usdc")
# Exports just the root layer content — not a flatten

02 — Stage Traversal

Stage traversal is how you programmatically walk through all prims in a scene — to collect data, validate assets, process geometry, or export content. It's a core skill for pipeline development and data exchange tools.

PythonAll traversal patterns you need to know
from pxr import Usd, UsdGeom

stage = Usd.Stage.Open("scene.usda")

# ── TRAVERSE ALL PRIMS (default — depth first) ───────────────────────
for prim in stage.Traverse():
    print(prim.GetPath(), prim.GetTypeName())
# Visits every prim in the stage in depth-first order
# Includes ALL prims — geometry, materials, lights, cameras

# ── TRAVERSE ONLY ACTIVE PRIMS ───────────────────────────────────────
for prim in stage.Traverse(Usd.PrimIsActive):
    print(prim.GetPath())
# Skips prims with active=false

# ── TRAVERSE ONLY MODEL PRIMS (by kind) ──────────────────────────────
for prim in stage.Traverse(Usd.PrimIsModel):
    print(prim.GetPath(), Usd.ModelAPI(prim).GetKind())
# Only visits assembly, group, component, subcomponent prims

# ── TRAVERSE ONLY MESH PRIMS ─────────────────────────────────────────
for prim in stage.Traverse():
    if prim.IsA(UsdGeom.Mesh):
        mesh = UsdGeom.Mesh(prim)
        pts = mesh.GetPointsAttr().Get()
        print(f"Mesh {prim.GetPath()} has {len(pts)} vertices")
# prim.IsA() checks if the prim is or inherits from a schema type

# ── TRAVERSE WITH PREDICATE COMBINATIONS ─────────────────────────────
import pxr.Usd as Usd
predicate = Usd.PrimIsActive & Usd.PrimIsLoaded
for prim in stage.Traverse(predicate):
    print(prim.GetPath())
# Only active AND loaded prims
# & = AND, | = OR, ~ = NOT

# ── GET PRIMS BY TYPE NAME (string-based) ────────────────────────────
meshes = [p for p in stage.Traverse() if p.GetTypeName() == "Mesh"]
lights = [p for p in stage.Traverse() if p.GetTypeName().endswith("Light")]

# ── TRAVERSE CHILDREN OF A SPECIFIC PRIM ────────────────────────────
root = stage.GetPrimAtPath("/World/Geometry")
for child in root.GetChildren():
    print(child.GetPath())   # direct children only

for desc in root.GetAllDescendants():
    print(desc.GetPath())    # all descendants recursively
Traversal predicates — common ones to know

Usd.PrimIsActive — skip inactive prims. Usd.PrimIsLoaded — skip unloaded payloads. Usd.PrimIsModel — only model hierarchy prims. Usd.PrimIsDefined — only prims with a def spec (skip overs). Combine with & (AND) and | (OR) for complex filters.

03 — USD Command-Line Tools

USD ships with a set of command-line tools that are essential for pipeline work and debugging. You need to know what each one does for the exam.

usdcat

Print or convert USD files. Text to binary, binary to text, flatten a stage. Most versatile tool.

usdchecker

Validate a USD file against best practices and rules. Reports errors and warnings. Used in pipeline QA.

usdview

Interactive viewer. Browse the scenegraph, inspect properties, view the LayerStack, switch variants.

usdzip

Create or inspect .usdz packages. Package a USD file with its dependencies into one archive.

usddiff

Diff two USD files or stages. Shows what changed between versions.

usdresolve

Resolve asset paths — shows where USD would look for a referenced file given the current resolver config.

CLIEssential command examples
# ── usdcat ───────────────────────────────────────────────────────────
usdcat scene.usda                          # print contents to stdout
usdcat scene.usda -o scene.usdc            # convert text → binary
usdcat scene.usdc -o scene_debug.usda      # convert binary → text (debug)
usdcat --flatten scene.usda -o flat.usda   # flatten + convert

# ── usdchecker ───────────────────────────────────────────────────────
usdchecker scene.usda                      # validate the file
usdchecker scene.usdz --arkit             # stricter AR/web validation
# Common checks: missing defaultPrim, missing extent,
# unresolved references, invalid material bindings, bad paths

# ── usdzip ───────────────────────────────────────────────────────────
usdzip -r delivery.usdz scene.usda         # package scene + all deps
usdzip -l delivery.usdz                    # list contents of a usdz

# ── usddiff ──────────────────────────────────────────────────────────
usddiff version_1.usda version_2.usda      # show what changed

05 — Writing a USD Exporter

An exporter translates data FROM a DCC's internal format INTO USD. This is exam topic 4.7/4.8. Here is the standard pattern every USD exporter follows:

PythonStandard USD exporter pattern — the complete structure
from pxr import Usd, UsdGeom, UsdShade, Gf, Sdf

def export_to_usd(dcc_scene, output_path):
    """
    Standard USD exporter pattern.
    dcc_scene = the DCC's internal scene object (Maya, Houdini etc.)
    output_path = where to write the USD file
    """

    # ── STEP 1: Create the stage ─────────────────────────────────────
    stage = Usd.Stage.CreateNew(output_path)

    # ── STEP 2: Set stage metadata ───────────────────────────────────
    UsdGeom.SetStageUpAxis(stage, UsdGeom.Tokens.y)
    UsdGeom.SetStageMetersPerUnit(stage, 0.01)  # centimetres
    stage.SetMetadata("timeCodesPerSecond", dcc_scene.fps)

    # ── STEP 3: Create the root prim ─────────────────────────────────
    root = UsdGeom.Xform.Define(stage, "/World")
    stage.SetDefaultPrim(root.GetPrim())
    Usd.ModelAPI(root).SetKind(Kind.Tokens.assembly)

    # ── STEP 4: Export geometry ──────────────────────────────────────
    for dcc_mesh in dcc_scene.meshes:
        path = f"/World/Geometry/{dcc_mesh.name}"
        mesh = UsdGeom.Mesh.Define(stage, path)

        # Write required mesh attributes
        mesh.CreatePointsAttr().Set(dcc_mesh.vertices)
        mesh.CreateFaceVertexCountsAttr().Set(dcc_mesh.face_vertex_counts)
        mesh.CreateFaceVertexIndicesAttr().Set(dcc_mesh.face_vertex_indices)
        mesh.CreateSubdivisionSchemeAttr().Set("none")

        # Compute and set extent
        extent = UsdGeom.Boundable.ComputeExtentFromPlugins(
            mesh, Usd.TimeCode.Default()
        )
        mesh.CreateExtentAttr().Set(extent)

        # Write UV coordinates if present
        if dcc_mesh.uvs:
            uv_pv = mesh.CreatePrimvar(
                "st", Sdf.ValueTypeNames.TexCoord2fArray, "faceVarying"
            )
            uv_pv.Set(dcc_mesh.uvs)

        # Write transform
        UsdGeom.XformCommonAPI(mesh).SetTranslate(dcc_mesh.position)

    # ── STEP 5: Export materials ─────────────────────────────────────
    for dcc_mat in dcc_scene.materials:
        mat_path = f"/World/Looks/{dcc_mat.name}"
        material = UsdShade.Material.Define(stage, mat_path)
        pbr = UsdShade.Shader.Define(stage, f"{mat_path}/PBRShader")
        pbr.CreateIdAttr("UsdPreviewSurface")
        pbr.CreateInput("diffuseColor", Sdf.ValueTypeNames.Color3f).Set(dcc_mat.color)
        pbr.CreateInput("roughness", Sdf.ValueTypeNames.Float).Set(dcc_mat.roughness)
        material.CreateSurfaceOutput().ConnectToSource(pbr.ConnectableAPI(), "surface")

    # ── STEP 6: Bind materials to meshes ────────────────────────────
    for dcc_mesh in dcc_scene.meshes:
        mesh_prim = stage.GetPrimAtPath(f"/World/Geometry/{dcc_mesh.name}")
        mat = UsdShade.Material.Get(stage, f"/World/Looks/{dcc_mesh.material.name}")
        UsdShade.MaterialBindingAPI.Apply(mesh_prim).Bind(mat)

    # ── STEP 7: Save ────────────────────────────────────────────────
    stage.Save()
    return output_path

06 — Asset Validators

A validator is a script or tool that inspects a USD asset and reports problems — missing required data, broken references, wrong conventions. Writing validators is exam topic 4.6. The built-in validator tool is usdchecker.

PythonWriting a custom validator for exported assets
from pxr import Usd, UsdGeom, UsdShade

def validate_usd_asset(file_path):
    """
    Validate a USD asset before delivery.
    Returns a list of error strings — empty list = valid.
    """
    errors = []
    stage = Usd.Stage.Open(file_path)

    # ── CHECK 1: defaultPrim is set ──────────────────────────────────
    root_layer = stage.GetRootLayer()
    if not root_layer.defaultPrim:
        errors.append("ERROR: No defaultPrim set — referenced asset won't resolve")

    # ── CHECK 2: metersPerUnit is set ────────────────────────────────
    if not stage.HasMetadata("metersPerUnit"):
        errors.append("WARNING: metersPerUnit not set — scale may be wrong")

    # ── CHECK 3: All meshes have extent ──────────────────────────────
    for prim in stage.Traverse():
        if prim.IsA(UsdGeom.Mesh):
            mesh = UsdGeom.Mesh(prim)
            if not mesh.GetExtentAttr().HasAuthoredValue():
                errors.append(f"ERROR: {prim.GetPath()} has no extent")

    # ── CHECK 4: All material bindings resolve ───────────────────────
    for prim in stage.Traverse():
        if prim.HasAPI(UsdShade.MaterialBindingAPI):
            bound, _ = UsdShade.MaterialBindingAPI(prim).ComputeBoundMaterial()
            if not bound:
                errors.append(f"ERROR: {prim.GetPath()} has binding but no material resolves")

    # ── CHECK 5: No unresolved references ───────────────────────────
    for layer in stage.GetLayerStack():
        for asset in layer.GetExternalReferences():
            import os
            if not os.path.exists(asset):
                errors.append(f"ERROR: Broken reference → {asset}")

    return errors

# Usage:
issues = validate_usd_asset("my_asset.usda")
if issues:
    print("Validation FAILED:")
    for e in issues: print(f"  {e}")
else:
    print("Asset is valid ✓")

07 — Round-Trip Pipelines

A round-trip pipeline means data can go OUT of USD into a DCC, be edited, and come back INTO USD without losing data. Perfect round-tripping is difficult — data loss during translation is a real challenge. Exam topic 4.4 and 7.4.

A round-trip pipeline — USD → DCC → USD with no data loss

Why round-trips are hard — data loss scenarios

Common data loss problems

  • Composition arcs (references, variants, sublayers) — most DCCs don't understand them
  • Multiple UV sets may map to wrong primvar names
  • Custom primvars not supported by the DCC
  • USD-specific metadata lost
  • upAxis mismatch (Y vs Z) causes geometry flipping
  • metersPerUnit mismatch causes scale issues

Round-trip best practices

  • Flatten before sending to DCC — remove composition complexity
  • Document your data mapping explicitly
  • Run validator on re-import to catch issues early
  • Keep USD-specific data in a separate override layer
  • Match upAxis and metersPerUnit on export
  • Test with a simple asset before full pipeline

08 — Data Exchange: glTF and Beyond

Data exchange is about moving data between USD and other 3D formats. The most commonly tested format in the exam is glTF — a widely used open format for 3D on the web and in games. Exam topic 4.1 asks you to convert USD to glTF and ensure fidelity.

Real-world analogy

glTF is like PDF for 3D. It's an open, portable, widely-supported format that works in browsers, game engines, and AR apps. USD is more like a full production file format — much more powerful but also more complex. Converting USD to glTF is like saving a complex InDesign document to PDF — you get a portable result but you lose some of the source structure.

09 — Conceptual Data Mapping Documents

A conceptual data mapping document is a logical blueprint that specifies how source schema attributes map to target USD schema attributes. It is created before writing any importer/exporter code. The exam tests exactly what belongs in this document and what does NOT.

✅ Essential contents

  • Attribute correspondences — source name → target name, data type, units
  • Transformation rules — how complex conversions work (e.g. scale units, axis flip)
  • Mandatory vs optional fields — which must be present for a valid exchange
  • Semantic alignment — what each attribute means in both schemas

❌ Does NOT belong here

  • Version histories — belongs in schema management, not mapping docs
  • Physical storage format — implementation concern, not logical mapping
  • Example payloads — belongs in testing/implementation docs
  • Build instructions or deployment steps
ExampleWhat a conceptual data mapping document entry looks like
# ── EXAMPLE MAPPING ENTRY ────────────────────────────────────────────
#
# Source (Maya): mesh.uvSet[0]  (float2, UV per face-vertex)
# Target (USD):  primvars:st    (texCoord2f[], faceVarying)
#
# Attribute correspondence: uvSet[0] → primvars:st
# Data type conversion:     float2 → texCoord2f
# Interpolation:            per-face-vertex → faceVarying
# Mandatory:                YES — required for texturing
# Transformation rule:      V axis inverted (Maya V=0 bottom, USD V=0 top)
#
# ── WHAT THIS IS NOT ────────────────────────────────────────────────
# NOT: "Maya schema version 2024.1" ← version history, wrong place
# NOT: "Store in .usdc binary format" ← storage detail, wrong place
# NOT: example mesh with UVs ← test payload, wrong place
#
# ── EXAM ANSWER PATTERN ─────────────────────────────────────────────
# Q28 correct items: attribute correspondences + types + units,
#                    transformation rules, mandatory/optional fields
# Q28 wrong item you chose: "version histories" → schema management

10 — Custom Importers: The Intermediate Representation Pattern

The exam consistently tests the correct architecture for a custom USD importer. The answer is always the same: parse to an intermediate representation first, then populate USD. This decouples the two concerns and is the professional standard.

Real-world analogy

Importing a proprietary file directly into USD is like trying to translate a book from Japanese to English by writing English sentences word-by-word as you read each Japanese word. You'll miss context and grammar. The correct approach is to first understand the full Japanese sentence (intermediate representation), then write the English sentence (USD prims). Two separate passes — parsing then writing.

PythonThe intermediate representation pattern — correct importer architecture
from pxr import Usd, UsdGeom, UsdShade

# ── CORRECT PATTERN: Parse → Intermediate → USD ──────────────────────

# STEP 1: Parse proprietary format into intermediate data structure
# This isolates file format parsing from USD API calls
def parse_proprietary_file(filepath):
    intermediate = {
        "meshes": [],
        "transforms": [],
        "materials": [],
    }
    # Read and parse the proprietary file format here
    # Validate, transform, convert units
    # Store in clean Python data structures
    return intermediate

# STEP 2: Populate USD prims from the intermediate representation
# This isolates USD API usage from file format specifics
def populate_usd_stage(intermediate, stage):
    for mesh_data in intermediate["meshes"]:
        mesh = UsdGeom.Mesh.Define(stage, mesh_data["path"])
        mesh.GetPointsAttr().Set(mesh_data["points"])
        mesh.GetFaceVertexCountsAttr().Set(mesh_data["face_counts"])
        mesh.GetFaceVertexIndicesAttr().Set(mesh_data["indices"])

# STEP 3: Wire it together
def import_file(filepath, output_usd):
    intermediate = parse_proprietary_file(filepath)
    stage = Usd.Stage.CreateNew(output_usd)
    populate_usd_stage(intermediate, stage)
    stage.Save()

# ── WHY THIS PATTERN ────────────────────────────────────────────────
# 1. Decouples parsing from USD API — each can be tested independently
# 2. Error handling is cleaner — validate data before committing to USD
# 3. Easier to debug — know whether bug is in parsing or USD population
# 4. Batch USD API calls — more efficient than interleaving with I/O

# ── WRONG APPROACHES ────────────────────────────────────────────────
#
# ❌ Implementing a custom UsdSchema to represent proprietary data
#    Custom schemas useful for EXTENDING USD, NOT required for importing
#    Q31: "not strictly necessary for importing proprietary formats"
#
# ❌ Writing USD ASCII files directly without the USD API
#    Error-prone, bypasses USD's validation and layering system
#

# ❌ Using UsdGeom namespace ONLY for all imports
#    Animation may need UsdSkel, materials need UsdShade
#    Not all data fits in UsdGeom

11 — Data Interchange Script Best Practices

The exam tested several data interchange principles that were missing from the original notes. These are the correct practices for writing scripts that exchange data between USD stages.

✅ Always do

  • Leverage USD's layering and composition model to combine data sources — never manually merge in scripts
  • Use the USD Python API to create and populate UsdPrims with schemas
  • Use relative paths or asset resolvers for all references
  • Respect USD schemas for attribute definitions and validation
  • Use attribute variability and time-sampling for animated data
  • Implement custom schemas for application-specific data

❌ Never do

  • Use absolute file paths — reduces portability
  • Flatten all data into one ASCII file — loses modularity and performance
  • Ignore USD schemas — breaks validation and interoperability
  • Embed raw binary blobs without schema definitions
  • Write USD ASCII manually without using the USD API
  • Embed all proprietary metadata as unregistered string attributes
PythonCorrect vs incorrect data interchange patterns
from pxr import Usd, UsdGeom

# ── CORRECT: Use composition model, not manual merging ───────────────
# Don't manually read two stages and copy data
# Instead: use sublayers or references to compose them
stage = Usd.Stage.CreateNew("combined.usda")
stage.GetRootLayer().subLayerPaths = ["./geometry.usda", "./materials.usda"]
# USD's composition handles the merge — one line, not hundreds

# ── CORRECT: Relative paths not absolute ────────────────────────────
# WRONG:  chair.GetReferences().AddReference("C:/project/chair.usda")
# RIGHT:  chair.GetReferences().AddReference("./assets/chair.usda")
# Relative paths work on any machine. Absolute paths break portability.

# ── CORRECT: Use USD API, not hand-written ASCII ─────────────────────
# WRONG:  open("scene.usda", "w").write('#usda 1.0\ndef Mesh...')
# RIGHT:  mesh = UsdGeom.Mesh.Define(stage, "/World/Chair")
#         mesh.GetPointsAttr().Set(points)
# API = validated, schema-compliant, correct. Hand-written = fragile.

# ── CORRECT: Time-sampling for animated data ─────────────────────────
# WRONG:  attr.Set(value)  ← static default, animation lost
# RIGHT:  for frame, value in animation_data:
#             attr.Set(value, time=frame)  ← animated correctly

# ── CORRECT: Custom schemas for proprietary data ─────────────────────
# WRONG:  prim.CreateAttribute("my_data", Sdf.ValueTypeNames.String)
#              .Set(json.dumps(proprietary_data))
# RIGHT:  use registered custom schema or proper metadata conventions
#         this ensures interoperability and semantic meaning

Exporter Hooks — Pre/Post Export

Exporter hooks are defined extension points in the export workflow where custom logic can be injected without modifying the core exporter code. This is the correct way to customise a USD exporter in a production pipeline.

🔑
The exam principle: hooks EXTEND the pipeline at defined points. They never replace or override the core export function. Same as schema inheritance — extend, never override.

Pre-export hook

Runs BEFORE core export. Use to prepare data, validate input, set upAxis/metersPerUnit, add pipeline metadata to the stage.

Post-export hook

Runs AFTER core export. Use to add custom attributes to exported prims, run usdchecker, register the asset in a pipeline database.

PythonPre/post-export hook pattern — correct approach
from pxr import Usd, UsdGeom, Sdf

def pre_export_hook(stage, export_options):
    # Runs BEFORE core export — prepare and configure
    UsdGeom.SetStageUpAxis(stage, UsdGeom.Tokens.y)
    UsdGeom.SetStageMetersPerUnit(stage, 0.01)
    stage.GetRootLayer().SetCustomLayerData({

Removing Proprietary Dependencies

When USD assets depend on proprietary shaders or plugins (Arnold, RenderMan, studio-specific C++ plugins), they cannot be used by other tools. The correct fix is to refactor into USD native schemas and shader definitions — not to encapsulate, reference around, or replace USD.

❌ Wrong approaches

  • Replace USD with proprietary formats — increases dependency
  • Reference proprietary files without conversion — dependency stays
  • Encapsulate without converting — long-term maintenance problem

✅ Correct approach

  • Refactor proprietary shaders → UsdPreviewSurface or MaterialX
  • Refactor proprietary schema types → registered custom USD schemas
  • Any USD-compliant tool can then read the assets
Why this matters for portability

A shader using info:id = "ArnoldStandardSurface" only works in Arnold. The same material using info:id = "UsdPreviewSurface" works in any USD-compliant renderer — Storm, RenderMan, Karma, Blender, Omniverse. Refactoring to USD native removes the vendor lock-in without losing the material data.

Flattening Best Practices

Flattening resolves composition arcs into a single self-contained layer. The exam tests exactly what flattening does and does NOT do.

Action Part of flattening? Why
Remove references/payloads → self-contained✅ YESAll arcs resolved, file needs no external dependencies
Merge all layers into one✅ YESThe point of flattening — multiple layers become one
Ensure unique prim paths, no namespace collisions✅ YESMerging assets can create collisions — must be resolved
Collapse variants to currently selected variant✅ YESOnly the active selection survives — switching is gone
Bake time-sampled data to static values❌ NOTime samples are PRESERVED. Baking is a separate explicit step.
Convert geometry to a single mesh type❌ NOGeometry types never change — flattening doesn't touch them
Preserve variant sets exactly as original❌ NOVariants collapse to selected. Sets are NOT preserved.
Critical exam trap — time samples

"Baking all time-sampled data into static values" is NOT part of flattening. Flattening PRESERVES animation time samples exactly as they were. Baking timeSamples to static defaults is a separate, deliberate optimisation operation. The flat file still has all your keyframes.

UI/UX for USD Pipelines

USD pipeline tools follow specific UI/UX design principles that match USD's modular, graph-based architecture. The exam tests which patterns are correct and which are anti-patterns.

✅ Good pipeline UI/UX

  • Node-based visual editor with drag-and-drop — matches USD's graph structure
  • Real-time feedback on execution status and errors
  • Context-sensitive help and tooltips for each node
  • Support for branching and merging node flows
  • Customisation options for different pipeline types

❌ Anti-patterns

  • Modal dialogs for every config change — disrupts workflow
  • Linear-only pipeline flows — USD branches and merges
  • No customisation — pipeline diversity requires flexibility
Why node-based specifically?

USD's composition model is a graph — layers reference layers, assets reference assets, variants branch and merge. A node-based editor reflects this graph structure. Each node is a stage or layer. Connections are composition arcs. The UI matches the mental model of the technology.

🔧
Build Configuration Management

USD pipeline builds require proper configuration management to be reproducible across machines, operating systems, and team members.

✅ Correct practices

  • Explicit build variants in config files — toggle features clearly
  • Centralised config management — version control build settings
  • Conditional logic in build scripts for different platforms
  • Use build cache — faster iteration, not slower
  • Document all environment variables

❌ Anti-patterns

  • Undocumented environment variables — breaks reproducibility
  • Hardcoding paths — fragile, not portable
  • Ignoring build cache to force clean rebuilds — wastes time
Exam trap — build cache

"Ignoring build cache mechanisms to always force a clean rebuild" is WRONG. Build caches detect unchanged components and skip rebuilding them. Ignoring them causes unnecessary long build times. In production USD builds that may take hours, this is a critical efficiency concern.

Pipeline Design Principles + Stage Caching

Two topics never previously covered: stage caching and the change notification system. Both appeared in exam Q62.

Stage caching — UsdStageCache

Opening and composing a USD stage is expensive. Stage caching stores previously opened stages in memory so they can be reused without full recomposition. Critical for interactive tools and large pipelines.

PythonUsdStageCache and the change notification system
from pxr import Usd, Sdf

# ── STAGE CACHING ─────────────────────────────────────────────────────
cache = Usd.StageCache()

with Usd.StageCacheContext(cache):
    stage = Usd.Stage.Open("scene.usda")
    # Stage is now cached
    # Subsequent Open("scene.usda") returns cached stage — no recomposition
# Avoids redundant opens and recompositions → performance improvement

# ── CHANGE NOTIFICATION SYSTEM ───────────────────────────────────────
# USD sends change notifications when the stage is modified
# Listeners only update what actually changed — not the whole scene
#
# Without notifications: any edit → full scene reprocess
# With notifications:    any edit → only affected prims reprocess
#
# SdfChangeBlock batches notifications for bulk operations
with Sdf.ChangeBlock():
    for i in range(10000):
        attr.Set(new_value[i], time=i)
# ONE notification after the block — not 10,000 — massive perf gain

# ── CORE PIPELINE DESIGN PRINCIPLES ─────────────────────────────────
#
# ✅ Strong layering and composition arcs — non-destructive, modular
# ✅ Schema-based validation — enforce data integrity pipeline-wide
# ✅ Stage caching — avoid redundant recomposition
# ✅ Change notification system — only update what changed
# ✅ USD native composition over external scripts for transformation
#
# ❌ Embed all data in one file — no modularity, no reuse
# ❌ Avoid variant sets — they are powerful tools, not complexity sources
# ❌ Rely solely on external scripts — ignores USD's native power

Day 7 key takeaways

1Use .usdc for heavy data (geometry, animation). Use .usda for lightweight interface layers and debugging. Use .usdz for final delivery. Never use .usda for large geometry in production.
2usdcat converts formats and flattens. usdchecker validates assets. usdzip creates/inspects .usdz packages. usddiff diffs two files. Know what each does.
3Stage traversal: stage.Traverse() visits all prims. Filter with predicates: Usd.PrimIsActive, Usd.PrimIsModel, Usd.PrimIsLoaded. Combine with & and |.
4Exporter pattern: create stage → set metadata → create root prim → export geometry → export materials → bind materials → validate → save.
5Validator pattern: traverse stage → check defaultPrim → check metersPerUnit → check each mesh has extent → check material bindings resolve → check references exist.
6Round-trip challenges: USD composition arcs (variants, references, sublayers) are lost when going through DCCs. Flatten before sending to DCC, validate on re-import.
7glTF mapping: Xform→node, Mesh→mesh primitive, UsdPreviewSurface→PBR metallic-roughness, primvars:st→TEXCOORD_0, timeSamples→animation. Variants and composition arcs have no glTF equivalent.
8DCC integration modes: Native USD (no translation), Plugin (translation at I/O), Standalone converter (external script). Each has different fidelity tradeoffs.
9Conceptual data mapping must be documented explicitly — "UV set map1 maps to primvars:st as faceVarying texCoord2f." Don't assume the mapping is obvious.
10upAxis and metersPerUnit are the two most common sources of scale/orientation bugs in data exchange. Always explicitly set both on export and check on import.
11Conceptual mapping docs contain: attribute correspondences (types + units), transformation rules, mandatory/optional fields. NOT version histories, NOT storage format details, NOT example payloads.
12Custom importer pattern — parse proprietary format to intermediate representation FIRST, then populate USD prims. Never write USD ASCII directly. Never use UsdGeom-only. Custom schema not required for importing.
13Data interchange scripts — use USD's layering/composition model to combine sources. Relative paths not absolute. USD API not hand-written ASCII. Schemas for all attributes. Time-sampling for animation.
14Exporter hooks — use pre/post hooks to extend export at defined points. Never override the main export function. Never modify USD files on disk after export.
15Proprietary dependencies — refactor to USD native (UsdPreviewSurface, custom USD schemas). Encapsulating without converting does not remove the dependency.
16Flattening preserves time samples — animation is NOT baked to static. Variants collapse to selected variant. Geometry types are never changed by flattening.
17Build configuration — explicit variants, centralised version-controlled config, conditional platform logic. Never hardcode paths. Never ignore build cache.
18Stage caching (UsdStageCache) avoids redundant recomposition. Change notification system limits updates to only changed elements. SdfChangeBlock batches notifications for bulk writes.
