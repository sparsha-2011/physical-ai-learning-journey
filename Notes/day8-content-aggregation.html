01 — What is Instancing?

Instancing is USD's memory-sharing mechanism for efficiently handling multiple copies of the same asset. Instead of duplicating the scenegraph hierarchy for every copy, USD creates ONE shared version called a prototype and all copies point to it.

Real-world analogy

Instancing is like a rubber stamp. You have one stamp (the prototype) and you can stamp it anywhere on the page as many times as you want. Each stamp mark (instance) looks identical but is positioned differently. You don't carve a new stamp for each mark — you reuse the same one. If you change the stamp design, every future stamp uses the new design. Without instancing, it would be like carving a new stamp for every single mark — wasteful and slow.

Without vs with instancing — scenegraph size comparison

USD provides two instancing implementations:

Scenegraph Instancing

Shares composed prim hierarchies. Each instance is an individual prim with instanceable=true. Supports per-instance overrides on the instanceable prim itself.

Best for: dozens to thousands of complex assets

Point Instancing

Uses UsdGeomPointInstancer schema. Stores positions, orientations, and scales as arrays. Designed for massive counts — millions of instances.

Best for: millions of simple objects (grass, rocks, particles)

02 — Scenegraph Instancing

Scenegraph instancing works by marking a prim as instanceable. USD then analyzes all instanceable prims that share the same composition arcs and creates ONE implicit prototype for all of them to share.

🔑
The key requirement: instanceable prims must share the same composition arcs — the same references, payloads, variants. USD computes an "instancing key" from the composition arcs. Prims with matching keys share one prototype. If two prims reference different assets they get different prototypes.
USDAScenegraph instancing — two robots sharing one prototype
#usda 1.0

def Xform "World"
{
    # Both robots reference the SAME asset AND have instanceable=true
    # → USD creates ONE prototype, both share it

    def Xform "Robot_01" (
        prepend references = @./robot.usda@
        instanceable = true    # ← the magic metadata
    )
    {
        # Can override properties on the instanceable prim itself
        double3 xformOp:translate = (-5, 0, 0)
        # CANNOT author opinions on children (/Robot_01/Arm etc.)
        # because the hierarchy is shared via the prototype
    }

    def Xform "Robot_02" (
        prepend references = @./robot.usda@
        instanceable = true
    )
    {
        double3 xformOp:translate = (5, 0, 0)
        # Same composition arcs → same instancing key → same prototype
    }
}
# Result: one __Prototype__ prim with 500 prims in it
# Robot_01 and Robot_02 are lightweight — they just reference the prototype
# Memory: 500 prims instead of 1000
PythonSetting instanceable and working with prototypes
from pxr import Usd, UsdGeom

stage = Usd.Stage.Open("forest.usda")

# ── SET instanceable ON EXISTING PRIMS ──────────────────────────────
robot1 = stage.GetPrimAtPath("/World/Robot_01")
robot1.SetInstanceable(True)
# USD immediately computes the instancing key and creates a prototype

robot2 = stage.GetPrimAtPath("/World/Robot_02")
robot2.SetInstanceable(True)
# Same reference + instanceable=true → same key → shares prototype

# ── CHECK IF A PRIM IS AN INSTANCE ──────────────────────────────────
print(robot1.IsInstance())          # True
print(robot1.IsInstanceable())      # True

# ── GET THE PROTOTYPE ────────────────────────────────────────────────
prototype = robot1.GetPrototype()
print(prototype.GetPath())
# → /__Prototype_1  (implicit path created by USD)
print(prototype.IsPrototype())      # True

# ── LIST ALL PROTOTYPES ON THE STAGE ────────────────────────────────
for proto in stage.GetPrototypes():
    print(proto.GetPath())
# One prototype per unique instancing key

# ── INSTANCE PROXIES — navigating into an instance ───────────────────
# You cannot directly access children of an instance prim
# because the hierarchy lives in the prototype, not in the instance
arm = stage.GetPrimAtPath("/World/Robot_01/Arm")
print(arm.IsInstanceProxy())
# True — /World/Robot_01/Arm is an instance proxy
# It "proxies" the corresponding prim in the prototype
# You can READ from it but not WRITE to it

# ── DE-INSTANCE when you need per-instance edits ────────────────────
robot1.SetInstanceable(False)
# Now Robot_01's hierarchy is no longer shared
# You can now make sparse overrides on /World/Robot_01/Arm etc.
# Robot_02 still uses the prototype
The key restriction — you cannot edit prototype descendants

When a prim is instanceable, its descendants (children, grandchildren) live in the prototype and are immutable from the instance. You can only make edits on the instanceable prim itself — transforms, variant selections, material bindings at the root level. To make per-instance edits to descendants you must de-instance that specific prim with SetInstanceable(False).

Nested instancing

An instanceable prim can contain other instanceable prims. This is called nested instancing and allows building up large aggregate assets from smaller reusable pieces.

USDANested instancing — a city block made of buildings made of windows
# Building references window.usda 20 times (instanceable)
# CityBlock references building.usda 10 times (instanceable)
# Result: 2 prototypes, massive memory saving

def Xform "CityBlock" (
    prepend references = @./building.usda@
    instanceable = true
)
{
    # building.usda internally has 20 instanceable window prims
    # → nested instancing: windows share a prototype INSIDE buildings
    # CityBlock instances share the building prototype
    # → two levels of sharing, enormous memory reduction
}

03 — Point Instancing

UsdGeomPointInstancer is a schema for placing millions of instances of prototype objects using compact array attributes. Instead of individual prims, everything is stored as large arrays — positions, orientations, scales, and which prototype each point uses.

Real-world analogy

Point instancing is like a spreadsheet of object placements. Each row is one instance. Columns are: which object, X position, Y position, Z position, rotation, scale. The renderer reads the spreadsheet and stamps each object at the specified position. It's incredibly compact — a forest of a million trees is just one million rows in a spreadsheet, not a million prim hierarchies.

PythonUsdGeomPointInstancer — placing 1000 trees efficiently
from pxr import Usd, UsdGeom, Gf, Vt
import random

stage = Usd.Stage.CreateNew("forest.usda")

# ── CREATE THE POINT INSTANCER ───────────────────────────────────────
instancer = UsdGeom.PointInstancer.Define(stage, "/World/Forest")

# ── DEFINE PROTOTYPES — what objects to stamp ────────────────────────
# Prototypes live as children under the instancer's "prototypes" scope
proto_scope = stage.DefinePrim("/World/Forest/Prototypes", "Scope")

# Prototype 0 — pine tree
pine = UsdGeom.Xform.Define(stage, "/World/Forest/Prototypes/PineTree")
pine.GetPrim().GetReferences().AddReference("./pine_tree.usda")

# Prototype 1 — oak tree
oak = UsdGeom.Xform.Define(stage, "/World/Forest/Prototypes/OakTree")
oak.GetPrim().GetReferences().AddReference("./oak_tree.usda")

# Tell the instancer which prims are its prototypes
instancer.GetPrototypesRel().AddTarget("/World/Forest/Prototypes/PineTree")
instancer.GetPrototypesRel().AddTarget("/World/Forest/Prototypes/OakTree")
# Index 0 = PineTree, Index 1 = OakTree

# ── PLACE 1000 INSTANCES ─────────────────────────────────────────────
N = 1000

# protoIndices — which prototype each instance uses
proto_indices = [random.randint(0, 1) for _ in range(N)]
# 0 = pine, 1 = oak — randomly mixed

# positions — where each instance goes
positions = [
    Gf.Vec3f(
        random.uniform(-100, 100),
        0,
        random.uniform(-100, 100)
    )
    for _ in range(N)
]

# scales — random variation so not all trees are identical size
scales = [
    Gf.Vec3f(random.uniform(0.8, 1.3),
             random.uniform(0.9, 1.5),
             random.uniform(0.8, 1.3))
    for _ in range(N)
]

# orientations — random Y rotation for variety
orientations = []
for _ in range(N):
    angle = random.uniform(0, 360)
    half  = angle * 0.00872665   # half angle in radians
    orientations.append(Gf.Quath(Gf.Vec4f(
        0, round(0.5 * angle / 360, 3), 0, 1
    )))

# ── SET THE ARRAYS ON THE INSTANCER ─────────────────────────────────
instancer.GetProtoIndicesAttr().Set(proto_indices)
# One int per instance — which prototype to use

instancer.GetPositionsAttr().Set(positions)
# One Vec3f per instance — world space position

instancer.GetScalesAttr().Set(scales)
# One Vec3f per instance — per-axis scale

stage.Save()
print(f"Created forest with {N} trees")
print(f"protoIndices length: {len(proto_indices)}")
print(f"positions length:    {len(positions)}")
print(f"scales length:       {len(scales)}")
# All three arrays must have the same length = number of instances
Required arrays on PointInstancer

protoIndices is the only truly required array — it identifies which prototype each point uses. positions is also required to place instances. orientations and scales are optional — if omitted, USD uses identity rotation and uniform scale of 1. All arrays that ARE authored must have the same length as the number of instances.

04 — Scenegraph vs Point Instancing

Both solve the same problem but for different scales and use cases. This comparison is directly testable on the exam.

Property Scenegraph Instancing Point Instancing
How enabledinstanceable = true metadataUsdGeomPointInstancer schema
ScaleDozens to thousandsMillions
Per-instance overridesYes — on the instanceable prim itselfNo — changes require full array reauthoring
Prototype locationImplicit __Prototype__ prims on stageExplicit children under instancer
TraversalInstances are regular prims — traversal worksInstances are array entries — not individual prims
Best use caseComplex assets (buildings, robots, vehicles)Simple repeated objects (grass, rocks, particles)
Memory saving mechanismShared scenegraph hierarchyCompact array storage
Can de-instance?Yes — SetInstanceable(False)No — must remove from array
When to use which

Use scenegraph instancing when you have hundreds of complex assets that may need individual overrides (move this one robot, change this one building's variant). Use point instancing when you have millions of simple repeated objects where every instance is essentially identical (a forest floor of grass blades, particle effects, scattered rocks).

05 — IsA Schemas

A schema is a blueprint that defines what a prim is and what properties it has. There are two types — IsA schemas and API schemas. Understanding the difference is directly tested in the exam.

An IsA schema (also called a typed schema) defines the fundamental type of a prim. Every prim can only have ONE IsA schema. It sets the typeName metadata and provides fallback values for all its defined attributes.

Real-world analogy

An IsA schema is like a job title. A person can only have one job title at a time — "Engineer" or "Designer" or "Manager." The title defines what they fundamentally ARE and what responsibilities (attributes) come with the role. You can't be both an Engineer and a Manager at the same time under the IsA system.

Concrete IsA schemas

Can be instantiated — create actual prims in the scene. Have both a name and a typeName in the schema definition.

  • UsdGeomMesh → def Mesh "Chair"
  • UsdGeomSphere → def Sphere "Ball"
  • UsdGeomXform → def Xform "World"
  • UsdShadeMaterial → def Material "Wood"

Abstract IsA schemas

Cannot be instantiated — serve as base classes. Provide only a name, no typeName. Exist to group related concrete schemas.

  • UsdGeomImageable — base for all renderable prims
  • UsdGeomGprim — base for all geometric prims
  • UsdGeomBoundable — base for prims with extents
  • UsdLuxLightAPI — base for all lights
PythonWorking with IsA schemas
from pxr import Usd, UsdGeom

stage = Usd.Stage.CreateInMemory()
sphere = UsdGeom.Sphere.Define(stage, "/MySphere")
prim = sphere.GetPrim()

# ── CHECK IsA TYPE ───────────────────────────────────────────────────
print(prim.IsA(UsdGeom.Sphere))       # True — it IS a Sphere
print(prim.IsA(UsdGeom.Gprim))        # True — Sphere inherits from Gprim
print(prim.IsA(UsdGeom.Imageable))   # True — all gprims are imageable
print(prim.IsA(UsdGeom.Mesh))         # False — a Sphere is not a Mesh

# ── GET THE TYPE NAME ────────────────────────────────────────────────
print(prim.GetTypeName())   # "Sphere"

# ── SCHEMA FALLBACKS WORK AUTOMATICALLY ─────────────────────────────
print(sphere.GetRadiusAttr().Get())   # 1.0 — schema fallback
# You never Set() radius but it returns 1.0 because the Sphere schema
# defines:  double radius = 1.0

06 — API Schemas

An API schema does NOT set the typeName — it adds additional capabilities to a prim that already has an IsA type. Think of it as "has a" rather than "is a." A prim can have multiple API schemas applied at once.

Real-world analogy

API schemas are like certifications or licences added to a job. Someone is still an "Engineer" (IsA) but they might also have a "Certified Safety Inspector" licence AND a "Project Management" certification. Each certification adds new capabilities without changing what they fundamentally are. Multiple certifications can be held simultaneously — unlike job titles.

Type Description Examples
Non-applied Just provides a Python/C++ API interface. Does not change the prim definition. Cannot tell if it's "applied". UsdModelAPI, UsdGeomPrimvarsAPI, UsdShadeConnectableAPI
Single-apply Applied once per prim. Recorded in apiSchemas metadata. Has a public Apply() method. Adds properties to the prim. MaterialBindingAPI, UsdGeomMotionAPI, UsdPhysicsRigidBodyAPI
Multiple-apply Can be applied multiple times with different instance names. Properties are prefixed with the instance name. UsdCollectionAPI (applied once per collection)
PythonWorking with API schemas
from pxr import Usd, UsdGeom, UsdShade

stage = Usd.Stage.CreateInMemory()
mesh = UsdGeom.Mesh.Define(stage, "/World/Chair")
prim = mesh.GetPrim()

# ── APPLY A SINGLE-APPLY API SCHEMA ─────────────────────────────────
binding_api = UsdShade.MaterialBindingAPI.Apply(prim)
# Apply() records "MaterialBindingAPI" in the prim's apiSchemas metadata
# In USDA: prepend apiSchemas = ["MaterialBindingAPI"]

# ── CHECK IF AN API SCHEMA IS APPLIED ────────────────────────────────
print(prim.HasAPI(UsdShade.MaterialBindingAPI))   # True
print(prim.HasAPI(UsdGeom.MotionAPI))             # False — not applied

# ── LIST ALL APPLIED API SCHEMAS ────────────────────────────────────
print(prim.GetAppliedSchemas())
# ['MaterialBindingAPI'] — everything that's been Apply()'d

# ── NON-APPLIED API (no Apply() needed) ──────────────────────────────
# UsdModelAPI, UsdGeomPrimvarsAPI etc. are used directly without Apply()
model_api = Usd.ModelAPI(prim)
model_api.SetKind("component")
# No Apply() call — non-applied APIs just wrap the prim directly
# prim.HasAPI(Usd.ModelAPI) would return False — it's non-applied

# KEY DISTINCTION:
# IsA  → prim.IsA(UsdGeom.Mesh)        → checks typeName
# API  → prim.HasAPI(MaterialBindingAPI)→ checks apiSchemas list

07 — Custom Schemas

Custom schemas let you extend USD's type system with your own prim types and API schemas — designed for your specific pipeline, industry, or application. This is exam domain 6 (Customising USD — 6%).

When to use custom schemas vs custom properties

Custom properties — use when

  • Prototyping quickly — no schema generation needed
  • Data is specific to one asset or pipeline tool
  • Don't need standardisation across teams
  • No fallback values needed
  • Simple — just CreateAttribute()

Custom schemas — use when

  • Need standardised types across many tools/teams
  • Need schema fallback values
  • Need prim.IsA() or prim.HasAPI() to work
  • Building a plugin or industry standard
  • Need generated C++/Python API for the properties

The schema generation workflow

Custom schemas are generated from a schema.usda definition file using the usdGenSchema tool. This produces C++ and Python code you can compile into a plugin.

USDAA custom IsA schema definition — a sensor prim for IoT/digital twin pipelines
# schema.usda — defines a custom SensorPrim type
#usda 1.0
(
    subLayers = [
        @usd/schema.usda@   # inherit from base USD schemas
    ]
)

over "GLOBAL" (
    customData = {
        string libraryName   = "acmeSensors"
        string libraryPrefix = "AcmeSensor"
    }
) {}

# ── CUSTOM IsA SCHEMA ───────────────────────────────────────────────
class Xform "TemperatureSensor" (
    customData = {
        string className = "TemperatureSensor"
    }
    doc = """A prim representing a physical temperature sensor."""
    inherits = </Xform>
)
{
    # These attributes get fallback values automatically
    float sensor:temperature = 20.0    (
        doc = "Current temperature reading in Celsius"
    )
    float sensor:minRange    = -40.0
    float sensor:maxRange    = 125.0
    string sensor:serialNumber = ""
    bool  sensor:isCalibrated  = false
}

# ── CUSTOM API SCHEMA ───────────────────────────────────────────────
class "MaintenanceAPI" (
    customData = {
        token apiSchemaType = "singleApply"
    }
    inherits = </APISchemaBase>
)
{
    # Can be applied to ANY prim type
    string maintenance:lastServiceDate = ""
    int    maintenance:serviceIntervalDays = 365
    string maintenance:technician = ""
}
PythonUsing custom schemas after generation (conceptual)
# After running usdGenSchema on schema.usda you get:
# - generated C++ header/source files
# - generated Python bindings
# - plugInfo.json for USD plugin registration

# Then in your pipeline code:
from pxr import Usd, AcmeSensors   # your generated module

stage = Usd.Stage.CreateNew("factory.usda")

# Create a prim using your custom IsA schema
sensor = AcmeSensors.TemperatureSensor.Define(stage, "/Factory/Sensor_001")

# Schema fallbacks work automatically
print(sensor.GetTemperatureAttr().Get())   # 20.0 — schema fallback

# Set real readings
sensor.GetTemperatureAttr().Set(72.4)
sensor.GetSerialNumberAttr().Set("SN-ACME-001")

# Apply the maintenance API to ANY prim
maint = AcmeSensors.MaintenanceAPI.Apply(sensor.GetPrim())
maint.GetLastServiceDateAttr().Set("2024-03-15")

# Type checking works properly
print(sensor.GetPrim().IsA(AcmeSensors.TemperatureSensor))  # True
print(sensor.GetPrim().HasAPI(AcmeSensors.MaintenanceAPI))  # True

stage.Save()

08 — AssetResolver

The AssetResolver (Ar) is the plugin system that controls how USD resolves file paths. When USD encounters an asset path like @./chair.usda@ or @assets://chair/v3/chair.usda@, it calls the AssetResolver to convert that identifier into an actual file path on disk.

Real-world analogy

The AssetResolver is like a company's internal URL shortener. When someone shares a link like go/project-brief the shortener knows to resolve that to the actual long URL https://docs.company.com/projects/q4-2024/brief-v3.pdf. The AssetResolver does the same for USD asset paths — it translates abstract pipeline identifiers into concrete file system paths.

PythonUsing and querying the AssetResolver
from pxr import Ar, Usd

# ── GET THE DEFAULT RESOLVER ─────────────────────────────────────────
resolver = Ar.GetResolver()
# Returns the currently registered resolver plugin
# Default = ArDefaultResolver (simple file system resolver)
# Studios replace this with custom resolvers that talk to asset DBs

# ── RESOLVE A PATH ───────────────────────────────────────────────────
resolved = resolver.Resolve("./chair.usda")
print(resolved)
# → "/home/studio/project/assets/chair.usda"  (absolute path)
# The resolver expands relative paths to absolute paths

# ── WHAT A CUSTOM RESOLVER CAN DO ───────────────────────────────────
# Instead of file paths, studios use asset identifiers like:
#   @assets://chair/approved/v3@
#   @sgtk:sg_asset_type=Prop,asset=Chair,version=latest@
#
# The custom resolver translates these to actual disk paths:
#   /mnt/nas/studio/assets/Prop/Chair/v003/chair.usda
#
# Benefits:
# - USD files use abstract IDs not hard disk paths
# - Assets can move between disks without breaking references
# - Resolver can query a database for "latest approved version"
# - Works across different studios/operating systems

# ── RESOLVE CONTEXT ──────────────────────────────────────────────────
# A ResolveContext carries per-session settings
# e.g. "resolve everything at version 3" or "use the production DB"
with Ar.ResolverContextBinder(Ar.ResolverContext()):
    # All asset resolution inside this block uses the bound context
    stage = Usd.Stage.Open("shot.usda")
    # The resolver uses context settings to find the right versions
Why the AssetResolver matters for the exam

The exam topic 6.3 asks about customising USD for specific pipelines. The AssetResolver is the primary extensibility point for asset management integration. You don't need to know how to implement one — you need to know WHAT it does: translates abstract asset identifiers (pipeline URIs, database IDs) into actual file paths, enabling studios to use their own asset management systems with USD.

Day 8 key takeaways

1Instancing shares scenegraph data across multiple copies — one prototype, many instances. Saves memory and improves load time for large scenes.
2Scenegraph instancing: set instanceable=true on prims sharing the same composition arcs. USD creates implicit __Prototype__ prims. Good for dozens to thousands of complex assets.
3Instanceable prims are mutable at the root (you can move/rotate them) but their descendants are immutable — the hierarchy is shared. De-instance with SetInstanceable(False) to make per-instance child edits.
4Nested instancing: instanceable prims can contain other instanceable prims — two levels of sharing, used for city-scale scenes (windows inside buildings inside blocks).
5Point instancing: UsdGeomPointInstancer stores millions of instances as arrays (protoIndices, positions, orientations, scales). Best for grass, rocks, particles — simple repeated objects at massive scale.
6IsA schemas (typed schemas) define what a prim IS — one per prim, sets typeName. Checked with prim.IsA(). Can be concrete (instantiable) or abstract (base class only).
7API schemas define what a prim HAS — multiple can be applied per prim. Single-apply (MaterialBindingAPI), multiple-apply (CollectionAPI), non-applied (ModelAPI). Checked with prim.HasAPI().
8Use custom properties for quick prototyping and pipeline-specific data. Use custom schemas when you need standardisation, fallback values, and generated APIs across teams and tools.
9Custom schemas are defined in schema.usda and generated into C++/Python code using usdGenSchema. The output is a USD plugin — registered so USD knows about your new types.
10The AssetResolver translates abstract asset identifiers into file paths. The default resolver uses file system paths. Custom resolvers connect to studio asset management databases — enabling abstract IDs, version queries, and cross-platform portability.
