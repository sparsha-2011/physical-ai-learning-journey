01 — What is Composition?

Composition is the process USD uses to combine multiple layers of scene description into one final, unified scene. Instead of storing everything in one giant file, USD lets you split your scene across many files — each one contributing different data — and then compose them together at runtime.

Real-world analogy

Imagine building a burger. The bun comes from one supplier, the patty from another, the lettuce from a farm, the sauce from a kitchen. Each ingredient is separate. The final burger is the composed result. In USD: the geometry comes from one file, the animation from another, the lighting from a third. The Stage is the assembled burger.

Why not just put everything in one file? Because real productions involve teams. An animator shouldn't need to edit the same file as a lighting artist. Composition lets each person work in their own file, and the stage assembles everyone's work together without conflicts.

Composition in a real production — multiple teams, one stage

The mechanism that makes this work is called a composition arc. USD has 6 composition arcs, and each one is a different way to connect layers together. You'll learn all of them today and in Day 3.

02 — Opinions & Value Resolution

Every value stored in a layer is called an opinion. If three layers all have an opinion about the position of a chair, USD must pick one winning value. The process of picking the winner is called value resolution.

Real-world analogy

Think of opinions like votes in an election — but not all votes are equal. A senior director's vote outweighs an intern's. In USD, the "senior director" is the layer that is listed highest (first) in the layer stack. Their opinion always wins. If they have no opinion on something, the next person down the chain gets their way, and so on.

Value resolution — who wins when multiple layers have opinions?
The golden rule of value resolution

When multiple layers have opinions on the same property, the strongest layer always wins. The strength of a layer is determined by where it sits in the layer stack — higher up (listed first) = stronger. All weaker opinions are silently ignored for that property.

If a layer has no opinion on a particular property, USD moves down the stack until it finds one. If nobody has an opinion, USD uses the fallback value defined by the prim's schema (its type definition). This is why a Sphere has a radius even if you never explicitly set one — the schema fallback kicks in.

⚡
03 — LIVERPS: The Strength Order

USD has 6 types of composition arcs. When multiple arcs all provide an opinion for the same property, USD needs to know which one wins. The answer is LIVERPS — a mnemonic for the strength ordering from strongest to weakest.

LIVERPS — strength order from strongest (top) to weakest (bottom)
Memory trick

Say it out loud: "Liver PS" — like you're telling someone their liver is a PlayStation. Silly, but it sticks. L-I-V-E-R-P-S. Local, Inherits, Variants, References, Payloads, Specializes. Note: some older docs show LIVRPS without the E — both are used, the order is what matters.

Why does LIVERPS matter in practice?

Here's the scenario you'll face in exam questions: a prim gets opinions from multiple places. You need to figure out which one wins. The answer is always: find all the opinions, identify which LIVERPS category each one falls into, the one highest in the list wins.

Most important for today

Local (L) always beats References (R) and Payloads (P). This means: if you write an opinion directly in your scene file (local), it will always override anything coming from a referenced asset. This is how you customise assets without modifying the original file — and it's exactly what Q2 in the sample questions tests.

04 — Sublayers

A sublayer is the simplest composition arc. It stacks one entire layer on top of another, merging their namespaces. All prims from both layers end up in the same scene. The layer listed first is stronger.

Real-world analogy

Sublayers are like transparencies stacked on an overhead projector. Each transparency has some content drawn on it. Stack them all and you see everything at once. If two transparencies have something drawn at the same spot, the one on top (strongest) is the one you see. Sublayers work the same way — same namespace, strength wins on conflicts.

Sublayers — all prims share the same namespace

When to use sublayers

Sublayers are for separating workstreams on the same scene. Use them when multiple people need to work on the same set of prims without stepping on each other:

✅ Good uses for sublayers

  • Animator works in anim.usda, modeller works in model.usda — same prims, different data
  • Lighting artist adds lights to lighting.usda without touching geometry
  • FX artist adds particles to fx.usda
  • Separating a shot into: layout / animation / fx / lighting workstreams

❌ Wrong uses for sublayers

  • Bringing in a chair asset — use a Reference instead
  • Adding the same asset multiple times — sublayering creates ONE instance, not multiples
  • Heavy geometry you want to load lazily — use a Payload
  • Assets from other teams/files that need their own namespace
Sublayers vs References — the core distinction

Sublayers are like INCLUDE — the whole layer's namespace merges in. Think "same scene, different concerns." References are like GRAFT — one specific prim from another file is attached at a chosen path. Think "bringing an asset INTO this scene."

USDASublayer syntax in a file
# shot_042.usda — the root layer everyone opens
#usda 1.0
(
    subLayers = [
        @./anim.usda@,       # ← FIRST = STRONGEST. Wins all conflicts.
        @./layout.usda@,     # ← second strongest
        @./model.usda@       # ← LAST = WEAKEST
    ]
    # The @ signs delimit file paths in USD. ./ means same folder.
)
PythonWorking with sublayers in code
from pxr import Usd

stage = Usd.Stage.Open("shot_042.usda")
root_layer = stage.GetRootLayer()

# ── SEE THE CURRENT SUBLAYERS ──────────────────────────────────────
print(root_layer.subLayerPaths)
# ['./anim.usda', './layout.usda', './model.usda']
# Index 0 = strongest. Last index = weakest.

# ── ADD A SUBLAYER AS WEAKEST ───────────────────────────────────────
root_layer.subLayerPaths.append("./fx.usda")
# append() adds to the END = weakest position

# ── ADD A SUBLAYER AS STRONGEST ────────────────────────────────────
root_layer.subLayerPaths.insert(0, "./director_overrides.usda")
# insert(0, ...) puts it FIRST = makes it the strongest

# ── MUTE A LAYER (temporarily disable it) ──────────────────────────
stage.MuteLayer("./layout.usda")
# Muted layers don't contribute ANY opinions.
# Great for debugging: "which layer is causing this value?"
# Mute layers one by one until the bug disappears → you found it.
stage.UnmuteLayer("./layout.usda")
# Re-enable it.

# ── LOOP THROUGH ALL LAYERS ─────────────────────────────────────────
for layer in stage.GetLayerStack():
    print(layer.identifier)
# Prints in strength order. Root layer is always first.

05 — References

A Reference is a composition arc that takes a prim from another USD file and grafts it into your scene at a specific path. It's how you reuse assets — a chair model, a tree, a spaceship — across many different scenes.

Real-world analogy

A reference is like a shortcut (symlink) on your computer. The chair model lives in one place — chair.usda. Fifty different scenes can all "point to" that file with a reference. If you update chair.usda, all fifty scenes automatically see the update. You're not copying data — you're linking to it.

References — one asset file, used in many scenes

Reference vs Sublayer — the crucial difference

This is one of the most tested distinctions on the exam. Here's the clearest way to think about it:

Sublayer

Whole-layer operation. The entire layer merges in. All prims share the same namespace. Like stacking two transparencies.

Use for: same prims, different contributors (anim layer on top of model layer).

Reference

Prim-level operation. One specific prim from another file is grafted in at a chosen path in YOUR hierarchy. Like cutting and pasting one branch of a tree.

Use for: bringing an asset into a scene (a chair, a car, a building).

USDA syntax for references

USDADifferent ways to write a reference
# Basic reference — uses chair.usda's defaultPrim automatically
def Xform "Chair" (
    prepend references = @./chair.usda@
    # 'prepend' = add to FRONT of the references list = strongest reference
    # No prim path specified = uses chair.usda's defaultPrim (e.g. /Chair)
)
{
    # LOCAL opinions here override anything from chair.usda
    # because Local (L) beats References (R) in LIVERPS
    double3 xformOp:translate = (3, 0, 0)   # move the chair
}

────────────────────────────────────────────

# Reference with explicit prim path
def Xform "Chair" (
    prepend references = @./furniture.usda@</Furniture/DiningChair>
    # The <> part specifies EXACTLY which prim to pull from the file
    # This overrides defaultPrim — targets /Furniture/DiningChair directly
)

────────────────────────────────────────────

# Multiple references on one prim
def Xform "Chair" (
    prepend references = [
        @./chair_base.usda@,        # FIRST in list = stronger reference
        @./chair_repositioned.usda@  # second = weaker reference
    ]
)
# When multiple references, first listed = strongest

prepend vs append — list editing

In USD, references are stored in a list. You can add to the front (prepend) or the back (append) of that list. This is called list editing.

prepend vs append on a references list
PythonWorking with references in code
from pxr import Usd, UsdGeom, Sdf

stage = Usd.Stage.CreateNew("scene.usda")
world = UsdGeom.Xform.Define(stage, "/World")

# ── DEFINE A PRIM AND ADD A REFERENCE ──────────────────────────────
chair_prim = stage.DefinePrim("/World/Chair")
refs = chair_prim.GetReferences()
# GetReferences() returns a UsdReferences object — the list manager

refs.AddReference("./chair.usda")
# Adds a reference using chair.usda's defaultPrim
# AddReference = prepend (adds to front = strongest)

refs.AddReference("./furniture.usda", "/Furniture/DiningChair")
# With explicit prim path — targets /Furniture/DiningChair

# ── ADD A LOCAL OVERRIDE ON THE REFERENCED PRIM ─────────────────────
xform_api = UsdGeom.XformCommonAPI(chair_prim)
xform_api.SetTranslate(Gf.Vec3d(3, 0, 0))
# This local opinion WINS over the referenced position (LIVERPS: L beats R)

# ── REMOVE A REFERENCE ─────────────────────────────────────────────
refs.RemoveReference("./chair.usda")

# ── CLEAR ALL REFERENCES ───────────────────────────────────────────
refs.ClearReferences()

stage.Save()

06 — Payloads

A Payload is almost identical to a Reference — it also grafts a prim from another file into your scene. The one critical difference: payloads are not loaded by default. They are lazy. You have to explicitly ask USD to load them.

Real-world analogy

A reference is like a Wikipedia article that opens all linked pages automatically when you load the page — even if you only wanted to read the first paragraph. A payload is like a YouTube video where the video doesn't download until you click play. You see the thumbnail (prim exists in the hierarchy) but the heavy data isn't loaded until you need it.

Payloads — open the stage fast, load heavy assets on demand

Reference vs Payload — side-by-side

Property Reference Payload
When is it loaded?Always — immediately when stage opensOnly when you call stage.Load(path)
Can you unload it?NoYes — stage.Unload(path)
USDA keywordprepend references = @...@prepend payload = @...@
LIVERPS strengthR — 4th strongestP — 5th (weaker than reference)
Use forSmaller assets, always-needed structureHeavy geometry, large scene assets
Prim visible when unloaded?N/A (always loaded)Yes — prim exists, data is not loaded
USDA + PythonPayloads in both formats
# ── USDA SYNTAX ─────────────────────────────────────────────────────
def Xform "City"
{
    def Xform "BlockA" (
        prepend payload = @./block_a.usda@
        # 'payload' keyword instead of 'references' — otherwise identical
    )
    { }
}

────────────────────────────────────────────

# ── PYTHON API ──────────────────────────────────────────────────────
from pxr import Usd

# Open stage WITHOUT loading any payloads (instant, no heavy data)
stage = Usd.Stage.Open(
    "city.usda",
    load=Usd.Stage.LoadNone   # ← key: LoadNone skips all payloads
)
# The prim /City/BlockA EXISTS in the hierarchy (you can see it),
# but block_a.usda's geometry has NOT been loaded into memory yet.

# Load just the one block you need
stage.Load("/City/BlockA")
# Now block_a.usda is composed in. Geometry is available.

# Unload it when you're done
stage.Unload("/City/BlockA")
# Memory freed. The prim still exists, data is gone.

# Load ALL payloads (default behaviour — same as not specifying load=)
stage2 = Usd.Stage.Open("city.usda", load=Usd.Stage.LoadAll)

07 — Layer Offsets & Time Shifting

A Layer Offset lets you shift or scale the time samples of a referenced or sublayered file — without touching the original. This is how exam topic 1.4 works: "create a scene with multiple elements referencing the same animation at different time-offsets."

Real-world analogy

Imagine you have a recording of a drummer playing one bar. You want three drummers playing the same bar, but each starting slightly later (like a round or a canon). You don't re-record three times — you take the one recording, play it three times with different start offsets. That's exactly what a layer offset does to USD animation.

USDAThree chairs, same animation, different start times
def Xform "World"
{
    def Xform "Chair_A" (
        prepend references = @./animated_chair.usda@ (
            offset = 0      # starts at frame 0 — original timing
            scale  = 1.0    # 1.0 = normal speed
        )
    ) { }

    def Xform "Chair_B" (
        prepend references = @./animated_chair.usda@ (
            offset = 10     # starts 10 frames LATER than Chair_A
            scale  = 1.0
        )
    ) { }

    def Xform "Chair_C" (
        prepend references = @./animated_chair.usda@ (
            offset = 0
            scale  = 0.5    # plays at HALF speed — 2× slower
        )
    ) { }
}
# Result: 3 chairs, 1 animation file, 3 different timings.
# No duplication of animation data!
PythonLayer offsets in the Python API
from pxr import Usd, Sdf

stage = Usd.Stage.CreateNew("scene.usda")
chair_b = stage.DefinePrim("/World/Chair_B")

chair_b.GetReferences().AddReference(
    "./animated_chair.usda",
    layerOffset=Sdf.LayerOffset(offset=10.0, scale=1.0)
    # offset=10.0 → shift animation 10 time codes later
    # scale=1.0   → normal speed (2.0 = double speed, 0.5 = half speed)
    # Formula: composedTime = (originalTime × scale) + offset
)
The formula to remember

composedTime = (originalTime × scale) + offset
So if original time = 12, offset = 10, scale = 1.0: composedTime = (12 × 1.0) + 10 = 22. The animation that was at frame 12 now appears at frame 22 in the composed scene.

08 — Tracing a Composed Scene (Exam Q2 Walkthrough)

This is the most important practical skill for the exam. You'll be shown 2–3 USDA files and asked what the final composed value of a property is. Here is a step-by-step method that works every time.

🎯
The 3-step method: (1) Find every opinion about the property. (2) Identify what LIVERPS category each opinion belongs to. (3) The opinion in the strongest category wins — that's your answer.

The exact Q2 scenario from the study guide

Three files. What is the final xformOp:translate on /World/Chair?

USDAchair_base.usda — the weakest reference
#usda 1.0
def Xform "Chair" {
    double3 xformOp:translate = (0, 0, 0)
    uniform token[] xformOpOrder = ["xformOp:translate"]
}
USDAchair_repositioned.usda — also a reference (weaker than base)
#usda 1.0
def Xform "Chair" {
    double3 xformOp:translate = (1, 0, 0)
}
USDAscene.usda — the root layer (strongest)
#usda 1.0
def Xform "World" {
    def Xform "Chair" (
        prepend references = [@./chair_base.usda@, @./chair_repositioned.usda@]
        # chair_base is FIRST in list = stronger of the two references
    )
    {
        double3 xformOp:translate = (0, 1, 0)
        # ← THIS is a LOCAL opinion (written directly in scene.usda)
    }
}

Now trace it using the 3-step method

1
Find ALL opinions on /World/Chair.xformOp:translate
Opinion A: (0, 1, 0) — written directly in scene.usda inside the Chair prim's { } block
Opinion B: (0, 0, 0) — comes from chair_base.usda via reference
Opinion C: (1, 0, 0) — comes from chair_repositioned.usda via reference
2
Identify the LIVERPS category of each opinion
Opinion A: (0, 1, 0) is written directly in scene.usda → this is a Local (L) opinion
Opinion B: (0, 0, 0) comes via a reference arc → this is a Reference (R) opinion
Opinion C: (1, 0, 0) also comes via a reference arc → also a Reference (R) opinion
3
Apply LIVERPS — strongest category wins
L beats R in LIVERPS. So Opinion A (0, 1, 0) wins immediately.
The two reference opinions (B and C) are both overridden — they don't matter.
Final answer: (0, 1, 0) — Answer C in the study guide ✓
Visual summary of the Q2 composition trace
Why opinions don't take effect — the most common exam debugging scenario

If you see a question like "why isn't the chair moving to the right position?" the answer is almost always one of these: (1) A stronger layer has a different local opinion that's overriding it. (2) The layer with your opinion was accidentally muted. (3) You authored your change to the wrong edit target (wrong layer). (4) You used over but there's no def for that prim in any other layer.

Day 2 key takeaways — commit these to memory

1Composition = how USD combines multiple files into one scene. Every value stored in a layer is called an opinion.
2Value resolution = when multiple layers have opinions, the strongest layer wins. Weaker layers are silently ignored for that property.
3LIVERPS = Local, Inherits, Variants, References, Payloads, Specializes. This is the strength order. Local always wins. Memorise it.
4Sublayers stack whole layers — same namespace, all prims merge. Listed first = strongest. Use for separating workstreams (anim, layout, fx).
5References graft one prim from another file into your scene. Always loaded. Use prepend references = @./file.usda@. Local opinions override referenced ones (L beats R).
6Payloads are like references but lazy. Open stage with LoadNone for speed, then stage.Load(path) only what you need. Use for heavy geometry.
7Layer offsets time-shift referenced animation. offset=10 = start 10 frames later. scale=0.5 = half speed. Formula: composedTime = (original × scale) + offset.
8To trace a composed scene: find all opinions → identify LIVERPS category → strongest wins. Always apply this 3-step method on exam questions.
9prepend adds to the front of a list (stronger). append adds to the back (weaker). This is list editing — you're not replacing the list, you're adding to it.
10If an opinion isn't taking effect: check layer strength, check for muted layers, check edit target, check that over has a matching def somewhere.
