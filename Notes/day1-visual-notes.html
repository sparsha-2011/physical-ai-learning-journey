01 — The Stage

The Stage is the top-level container for your entire 3D scene. It's what your renderer, viewport, and tools actually look at. Everything — geometry, lights, cameras, materials — lives on the stage.

Real-world analogy

Think of a stage like the final printed poster that results from stacking multiple transparent sheets of acetate on top of each other. Each sheet has some content on it. The poster you see is all the sheets composed together. In USD, those sheets are called Layers.

Here's the key insight that confuses beginners: a Stage is not a file. It's a live, in-memory composition of one or more files (layers). When you open a USD file, USD assembles a Stage from that file and any other files it references.

How a Stage is assembled from Layers

Creating and opening stages in Python

Python Three ways to create a stage 
from pxr import Usd

# ── WAY 1: Create a stage saved to a real file ──────────────────────
stage = Usd.Stage.CreateNew("my_scene.usda")
# The file is created on disk IMMEDIATELY, even before you add anything.
# Open it in a text editor right now and you'd see just:  #usda 1.0


# ── WAY 2: Open an existing file ────────────────────────────────────
stage = Usd.Stage.Open("my_scene.usda")
# Loads the file and assembles all its layers into one stage.
# If the file references other files, those are loaded too.


# ── WAY 3: Create a stage with NO file (in-memory only) ─────────────
stage = Usd.Stage.CreateInMemory()
# Nothing is written to disk. Great for testing and temporary work.
# To save it later: stage.Export("output.usda")


# ── SAVING ──────────────────────────────────────────────────────────
stage.Save()
# Writes all changes back to the file(s) on disk.

# ── INSPECT AS TEXT (very useful for debugging!) ─────────────────────
print(stage.ExportToString(addSourceFileComment=False))
# Prints the USDA content of the stage as text.
# This is EXACTLY what the exam shows you in code questions.
Exam tip

CreateInMemory() is the answer when a question asks about creating a stage with no file on disk. CreateNew() always creates a file immediately.

02 — Layers

A Layer is a single USD file (or in-memory buffer) that contains some scene description. It's one "sheet" of data. Layers are the physical things that exist on disk. Stages are the assembled result of combining layers.

Real-world analogy

Layers are like tracks in a music DAW (like GarageBand or Pro Tools). The drums are on one track, bass on another, vocals on a third. Each track exists independently. The final song you hear is all tracks playing together — that's the Stage. Just like audio tracks, the order matters: a louder track can "drown out" a quieter one.

Every layer contains opinions — its own values for prims and properties. When a Stage is opened, USD looks at all opinions from all layers and picks the winning value for each property. The layer listed highest (first) in the stack is the strongest — its opinions win.

USDA A stage that combines three layers 
# scene.usda  ← this is the ROOT LAYER (the file you open)
#usda 1.0
(
    subLayers = [
        @./anim.usda@,      # ← FIRST = STRONGEST. Wins all conflicts.
        @./layout.usda@,    # ← second strongest
        @./models.usda@     # ← LAST = WEAKEST. Overridden by everything above.
    ]
)
# The @ signs are USD's way of writing file paths.
# The ./ means "same folder as this file".
💡
Key rule: If two layers both have an opinion about the same value (e.g. the X position of a chair), the layer listed first (higher up in subLayers) always wins. This is called strength ordering and it's the foundation of everything in USD composition.
Python Working with layers in code 
from pxr import Usd, Sdf

stage = Usd.Stage.Open("scene.usda")

# ── GET THE ROOT LAYER ──────────────────────────────────────────────
root_layer = stage.GetRootLayer()
# The root layer = the file you opened. It's the "anchor" of the stage.
print(root_layer.identifier)  # prints the file path

# ── SEE ALL LAYERS IN STRENGTH ORDER ───────────────────────────────
for layer in stage.GetLayerStack():
    print(layer.identifier)
# Prints layers strongest-first. Root layer is always first.

# ── ADD A SUBLAYER ─────────────────────────────────────────────────
root_layer.subLayerPaths.append("./fx.usda")
# append() = add to END = becomes the WEAKEST sublayer

root_layer.subLayerPaths.insert(0, "./overrides.usda")
# insert(0, ...) = add to FRONT = becomes the STRONGEST sublayer

# ── EDIT TARGET: which layer receives your edits ────────────────────
print(stage.GetEditTarget().GetLayer().identifier)
# By default = root layer. All your Set() calls go here.
# You can change it to write to a different layer:
stage.SetEditTarget(stage.GetLayerStack()[1])
# Now edits go to the SECOND layer, not the root.
# Getting this wrong is one of the most common bugs in USD pipelines!

03 — Prims

A Prim (short for Primitive) is every individual object in your scene. Every mesh, every light, every camera, every material, every empty organising container — all prims. The word comes from Pixar's original USD design and just means "a node in the scene graph."

Real-world analogy

Think of your scene graph like a file system on your computer. Prims are the folders and files. Some prims are like folders (they contain other prims but hold no geometry themselves — like Xform and Scope). Some prims are like files with data (they contain the actual 3D content — like Mesh, Sphere, Material).

A typical scene hierarchy — every node is a prim

The three prim specifiers — you must know these!

Every prim in a USDA file starts with one of three keywords. These appear in almost every exam question that shows code.

def

Define — creates a brand new prim. This is the normal one you use when you're the author. It claims full ownership of this prim.

over

Override — adds opinions on top of a prim that already exists in another layer. It doesn't create the prim — it just modifies it.

class

Class — a template prim. Never rendered directly. Used with the inherits arc so other prims can inherit from it.

Common exam trap

An over on its own does nothing if there is no def for the same prim in another layer. The over just silently has no effect. This is a very common debugging scenario — "why isn't my opinion taking effect?"

USDA def vs over in two different layers 
# model.usda  (weaker layer — loaded second)
def Xform "Chair"          # def = I am DEFINING this prim
{
    double3 xformOp:translate = (0, 0, 0)   # original position
}

────────────────────────────────────────────

# anim.usda  (stronger layer — loaded first)
over "Chair"                # over = I am OVERRIDING this prim
{                            # Note: NO type (Xform) with 'over'!
    double3 xformOp:translate = (5, 0, 0)   # new position (wins!)
}
# Result: chair sits at (5,0,0) in the final scene

Python Creating and navigating prims 
from pxr import Usd, UsdGeom

stage = Usd.Stage.CreateInMemory()

# ── DEFINE A PRIM (basic way) ───────────────────────────────────────
chair = stage.DefinePrim("/World/Chair", "Xform")
# First arg = the path.  Second arg = the type.
# Produces:  def Xform "Chair" { }
# Note: USD automatically creates /World if it doesn't exist!

# ── DEFINE A PRIM (schema-specific, preferred way) ──────────────────
sphere = UsdGeom.Sphere.Define(stage, "/World/MySphere")
# UsdGeom.Sphere is the Python class for Sphere prims.
# This way gives you type-safe access to Sphere's specific attributes.
# It's preferred because you can then call sphere.GetRadiusAttr() etc.

sphere.CreateRadiusAttr().Set(5.0)
# Sets radius = 5 on the sphere. In USDA:  double radius = 5

# ── GET A PRIM BY PATH ──────────────────────────────────────────────
prim = stage.GetPrimAtPath("/World/Chair")
# Returns the prim at that exact path.
# If the path doesn't exist, returns an INVALID prim (evaluates as False)
if not prim:
    print("Prim not found!")

# ── GET TYPE NAME ───────────────────────────────────────────────────
print(prim.GetTypeName())   # → "Xform"
# If the prim was defined with 'over' (no type), returns "" empty string

# ── NAVIGATE THE HIERARCHY ─────────────────────────────────────────
for child in prim.GetChildren():
    print(child.GetPath())    # prints paths of direct children only

04 — Properties

Properties are the actual data stored on a prim. A Sphere prim without a radius property would just be an empty shell. Properties are what make prims meaningful.

There are exactly two types of properties in USD:

Attribute
Stores a typed value — a number, a colour, a vector, a string. This is where your actual 3D data lives.
double radius = 5.0
color3f[] primvars:displayColor
Relationship
Stores a path pointing to another prim. Doesn't store values — stores connections.
rel material:binding =
</Materials/Metal>

Attribute value types — the ones you'll see in the exam

What is a primvar?

Attributes that start with primvars: are special — they are Primitive Variables that get inherited down to child prims and are used by renderers for shading. The most important ones to know: primvars:displayColor (a quick preview colour) and primvars:displayOpacity.

Python Reading and writing attributes 
from pxr import Usd, UsdGeom, Gf

stage = Usd.Stage.CreateInMemory()
sphere = UsdGeom.Sphere.Define(stage, "/World/Ball")
cube   = UsdGeom.Cube.Define(stage, "/World/Box")

# ── SET AN ATTRIBUTE VALUE ──────────────────────────────────────────
sphere.GetRadiusAttr().Set(5.0)
# GetRadiusAttr() returns the Attribute OBJECT (not the value yet!)
# .Set(5.0) writes the value 5.0 to it
# In USDA this becomes:   double radius = 5

# ── GET AN ATTRIBUTE VALUE ──────────────────────────────────────────
radius = sphere.GetRadiusAttr().Get()
print(radius)   # → 5.0
# .Get() with no arguments returns the DEFAULT value.
# .Get(time=12) would return the value at time code 12 (for animation).

# ── SET A COLOUR ────────────────────────────────────────────────────
sphere.GetDisplayColorAttr().Set([(1.0, 0.0, 0.0)])
# Note: wrapped in a LIST because displayColor is color3f[] (an array)
# (1.0, 0.0, 0.0) = red.  (0.0, 1.0, 0.0) = green.  (0.0, 0.0, 1.0) = blue

# ── SET A POSITION ──────────────────────────────────────────────────
UsdGeom.XformCommonAPI(sphere).SetTranslate(Gf.Vec3d(0, 5, 0))
# UsdGeom.XformCommonAPI is a helper that manages transform attributes.
# Gf.Vec3d is a 3D vector from USD's math library.
# This creates:  double3 xformOp:translate = (0, 5, 0)
#                uniform token[] xformOpOrder = ["xformOp:translate"]

# ── SET THE EXTENT (bounding box) ───────────────────────────────────
cube.GetExtentAttr().Set([(-2,-2,-2), (2,2,2)])
# Extent = the axis-aligned bounding box of the prim.
# IMPORTANT: USD does NOT auto-update extent when you change geometry!
# If you change the cube's size, you MUST manually update extent too.
# Forgetting this is a common bug (exam topic 5.6).

# ── MATERIAL BINDING (a Relationship, not an Attribute) ─────────────
from pxr import UsdShade
material = UsdShade.Material.Define(stage, "/Materials/MetalMat")
UsdShade.MaterialBindingAPI(sphere.GetPrim()).Bind(material)
# This creates a RELATIONSHIP (not an attribute) on the sphere prim:
#   rel material:binding = </Materials/MetalMat>
# It points the sphere TO the material. That's what a relationship does.

05 — Prim & Property Paths

Every prim and property in USD has a unique address called a path. Think of it exactly like a file path on your computer — it tells you where something lives in the hierarchy.

Anatomy of a USD path
Examples Reading USD paths 
# PRIM paths — the address of a prim
/World                          # prim at root level, named "World"
/World/Chair                    # prim "Chair" inside "World"
/World/Chair/seat_geo           # prim "seat_geo" inside "Chair"
/World/Materials/MetalMat       # prim nested 2 levels deep

# PROPERTY paths — prim path + dot + property name
/World/Chair.visibility         # the 'visibility' attribute on Chair
/World/Chair.xformOp:translate  # the translate attribute (colon in name = namespace)
/World/Ball.primvars:displayColor  # a primvar attribute

# PATH RULES:
# - Always starts with /
# - Prim names separated by /
# - Property separated from its prim by a dot (.)
# - Colons (:) in property names = namespacing (e.g. xformOp:translate)
# - No spaces allowed anywhere in a path

06 — USD File Formats

USD has three file formats. Knowing the differences is directly tested in exam topic 4.3 and implicitly in topic 1.9. Memorise this table.

.usda
USD ASCII
Human-readable text. You can open it in any text editor and read it. Great for learning, debugging, and version control. Slow to parse, large file size.
.usdc
USD Crate (binary)
Compact binary format. Very fast to load. Cannot be read in a text editor. This is what production pipelines use for performance.
.usdz
USD Zip Archive
A zip file containing a USD file plus all its dependencies (textures, sublayers). Self-contained — no broken references. Used for delivery to external parties.
Exam trap — .usdz for external delivery

Exam topic 1.9 asks how to "prepare an internal asset to be delivered to external parties." The answer is .usdz — because it bundles everything into one file so the recipient can't have missing textures or broken references. .usda and .usdc can both have references to other files that the recipient might not have.

Property .usda .usdc .usdz
Human-readable?✅ Yes❌ No❌ No
Load speedSlowVery fastFast
File sizeLargeSmallCompressed
Good for debugging?✅ Yes❌ No❌ No
Good for production?❌ No✅ YesDelivery only
Self-contained?❌ No❌ No✅ Yes
Can reference external files?✅ Yes✅ YesRead-only archive
Python Converting between formats 
from pxr import Usd

# Open a text (usda) stage and save it as binary (usdc)
stage = Usd.Stage.Open("scene.usda")
stage.Export("scene.usdc")
# The file extension in the Export() path determines the output format.
# USD automatically encodes to the correct format.

# Flatten a complex multi-layer stage to a single flat file
flat_layer = stage.Flatten()
flat_layer.Export("flattened.usda")
# Flatten() bakes ALL composition arcs into one layer.
# No references, no sublayers — just pure baked data.
# Use this when delivering to an external studio (exam topic 1.9).

07 — Metadata

Metadata is data about data. It doesn't describe 3D content (like geometry or colour) — it describes properties of the layer itself or of a specific prim. In USDA files, metadata lives inside ( ) parentheses blocks.

Real-world analogy

Metadata is like the label on a paint tin. The label says "this is red paint, for indoor use, 2.5 litres" — it describes the tin, not the painting you'll make. Similarly, USD metadata says "this file uses centimetres, Y is up, the main prim is /World" — it describes the file, not the 3D scene.

USDA Stage-level and prim-level metadata 
#usda 1.0
(
    # ── STAGE-LEVEL METADATA (describes this whole layer) ──────────────

    defaultPrim = "World"
    # When another file references THIS file, it uses /World as the target.
    # Without this, the referencing file has to specify a prim path manually.
    # This is what exam Q1 is testing!

    metersPerUnit = 0.01
    # 0.01 = centimetres (1 USD unit = 1 cm)
    # 1.0  = metres
    # USD does NOT auto-convert units. This is metadata only — a label.
    # If you reference a metre file into a centimetre stage,
    # the asset will appear 100x too small (exam Q11).

    upAxis = "Y"
    # Which axis points "up". "Y" = Maya default. "Z" = Blender/Houdini.
    # Again, USD doesn't rotate things — this is a hint for tools.

    timeCodesPerSecond = 24
    # 24 time codes = 1 second. So time code 48 = 2 seconds.
    # Change this for different frame rates (30fps, 60fps etc.)
)

def Xform "World" (
    # ── PRIM-LEVEL METADATA (describes just this prim) ─────────────────

    kind = "assembly"
    # Classifies this prim in the model hierarchy.
    # Values: "assembly", "component", "group", "subcomponent"

    doc = "The root world prim for this shot"
    # A human-readable description. Shows up in USD tools.

    active = true
    # active=false means this prim is excluded from the scene entirely.
    # It becomes invisible AND is skipped during traversal.
)
{
    # Attributes go in the curly braces { }, not in parentheses ( )
}
🔑
Key visual trick: In USDA files, ( ) parentheses always contain metadata. { } curly braces always contain attributes and child prims. Learn to spot this instantly — it tells you what kind of data you're looking at.
Python Setting metadata in code 
from pxr import Usd, UsdGeom, Kind

stage = Usd.Stage.CreateNew("scene.usda")

# ── STAGE-LEVEL METADATA ────────────────────────────────────────────
UsdGeom.SetStageUpAxis(stage, UsdGeom.Tokens.y)      # set Y-up
UsdGeom.SetStageMetersPerUnit(stage, 0.01)           # centimetres
stage.SetMetadata("timeCodesPerSecond", 24)           # 24fps

# ── SET defaultPrim ─────────────────────────────────────────────────
world = UsdGeom.Xform.Define(stage, "/World")
stage.SetDefaultPrim(world.GetPrim())
# Now when another file references this file, /World is used automatically.

# ── PRIM-LEVEL METADATA ─────────────────────────────────────────────
prim = stage.GetPrimAtPath("/World")
Usd.ModelAPI(prim).SetKind(Kind.Tokens.assembly)    # set kind
prim.SetMetadata("documentation", "Root world prim") # set doc string

# ── READ METADATA ───────────────────────────────────────────────────
root_layer = stage.GetRootLayer()
print(root_layer.defaultPrim)       # → "World"
print(root_layer.metersPerUnit)     # → 0.01

08 — Time Codes & Animation

USD handles animation by storing multiple values for one attribute at different points in time. Each point in time is called a time code (think of it as a frame number).

Real-world analogy

A time sample is like a keyframe in traditional animation software. In Maya or Blender, you set "at frame 0 the ball is here, at frame 24 it's over there." USD does exactly the same thing but calls them time codes instead of frames, and stores them in a special syntax called timeSamples.

USDA An animated bounce — ball moves and comes back 
#usda 1.0
(
    timeCodesPerSecond = 24   # 24 time codes = 1 second
)

def Xform "Ball"
{
    double3 xformOp:translate.timeSamples = {
        # time code : value at that time
        0:  (0, 0, 0),    # at frame 0:  ball is at the origin
        12: (0, 5, 0),    # at frame 12: ball is 5 units in the air
        24: (0, 0, 0),    # at frame 24: ball is back at origin (1 second later)
    }
    uniform token[] xformOpOrder = ["xformOp:translate"]
    # 'uniform' means this attribute does NOT animate (same for all time).
    # The xformOpOrder tells USD which transform ops to apply and in what order.
}
Python Setting and reading animated values 
from pxr import Usd, UsdGeom, Gf

stage = Usd.Stage.CreateNew("anim.usda")
stage.SetMetadata("timeCodesPerSecond", 24)

ball = UsdGeom.Xform.Define(stage, "/Ball")

# ── CREATE THE TRANSLATE ATTRIBUTE WITH TIME SAMPLES ────────────────
translate_attr = ball.AddTranslateOp().GetAttr()
# AddTranslateOp() sets up the translate xform operation.
# .GetAttr() gets the underlying Usd.Attribute object so we can set values.

translate_attr.Set(Gf.Vec3d(0, 0, 0),  time=0)
# time=0 makes this a TIME SAMPLE at frame 0, not a default value.

translate_attr.Set(Gf.Vec3d(0, 5, 0),  time=12)
# At frame 12, ball is 5 units up.

translate_attr.Set(Gf.Vec3d(0, 0, 0),  time=24)
# At frame 24, back to origin.

# ── READ A VALUE AT A SPECIFIC TIME ─────────────────────────────────
val = translate_attr.Get(time=12)
print(val)   # → Gf.Vec3d(0, 5, 0)
# Between keyframes, USD interpolates automatically.

# ── READ THE DEFAULT (non-animated) VALUE ───────────────────────────
default_val = translate_attr.Get()
# Returns the default value (if one was set), or None if only timeSamples exist.
# .Get() with no argument = default value, not time code 0!

stage.Save()
Key distinction

attr.Get() returns the default value (authored without a time code). attr.Get(time=0) returns the value at time code 0. These can be different things! The default value is what you get when there are no time samples at all.

Day 1 key takeaways — commit these to memory

1A Stage is the assembled scene — not a file. A Layer is the actual file. Stages are composed from one or more layers.
2Layers are ordered by strength. The layer listed first in subLayers is the strongest — its opinions win over all others.
3def creates. over overrides (no type allowed). class templates. An over with no matching def in another layer does nothing.
4Properties are either Attributes (typed values like numbers and colours) or Relationships (paths pointing to other prims, like material bindings).
5Prim paths use / separators. Property paths add a . after the prim path. Namespace separators inside property names use :.
6.usda = readable text (debugging). .usdc = fast binary (production). .usdz = portable zip for external delivery.
7Metadata lives in ( ) parentheses. Attributes live in { } curly braces. defaultPrim and metersPerUnit are the most exam-critical metadata fields.
8USD does NOT auto-convert units. metersPerUnit is just a label. A metres asset in a centimetres stage appears 100× too small.
9Animation = timeSamples on an attribute. attr.Get(time=12) reads at frame 12. attr.Get() reads the default value — not the same thing!
10The edit target decides which layer receives your edits. Accidentally authoring to the wrong layer is one of the most common USD pipeline bugs.
