01 — Why Custom Schemas Exist

Before getting into how custom schemas work, it's critical to understand why they exist — because the exam tests whether you understand the PURPOSE, not just the mechanics.

Real-world analogy

USD ships with schemas for common 3D concepts — Mesh, Sphere, Material, Light. These cover general-purpose use cases. But a factory automation company needs prims for "TemperatureSensor" and "ConveyorBelt." A film studio needs prims for "CharacterRig" and "ShotCamera." These don't exist in USD. Custom schemas let you ADD these as first-class USD citizens — with all the type safety, API generation, and tool integration of built-in schemas.

Use custom schemas when you need

  • Standardised types used across many tools and teams
  • Schema fallback values (defaults when nothing is authored)
  • prim.IsA() or prim.HasAPI() to work correctly
  • Generated C++ and Python API for properties
  • Full integration with USD's type system and tools
  • Custom validation rules on prim structure

Do NOT use custom schemas when

  • You just need a few pipeline-specific attributes quickly
  • The data is one-off and not shared across teams
  • You don't need fallback values or type checking
  • Prototyping — use custom properties first, schema later

02 — The Inheritance Hierarchy: Who to Inherit From

This is the exact thing the exam Q25 tested. There are multiple schema base classes and choosing the WRONG one was a common exam trap. Here's the full picture.

Real-world analogy

The schema hierarchy is like an employee org chart. UsdSchemaBase is the CEO — everyone technically reports to them but you never hire someone who reports DIRECTLY to the CEO without going through the right management chain. UsdTyped is the VP of Scene Objects. UsdGeomImageable is the Manager of Renderable Objects. You hire new people (schemas) at the right level for their job — not all the way at the top.

USD Schema Inheritance Hierarchy

The three inheritance choices — and when to use each

Base class What it gives you When to use Exam answer?
UsdSchemaBase The absolute root. Requires implementing EVERYTHING manually including type registration, property management, and all API plumbing. Almost never directly. Only if building completely new schema infrastructure. ❌ Wrong answer — error-prone, not recommended
UsdTyped Sets typeName on the prim. All the plumbing for IsA() to work. Schema fallback values. The standard base for typed (IsA) schemas. Any custom IsA schema that is a new "type" of prim. TemperatureSensor, ConveyorBelt, CharacterRig. ✅ Correct — recommended standard base
UsdGeomImageable Everything UsdTyped gives PLUS visibility, purpose, and all rendering infrastructure. Inherits from UsdTyped already. Custom schemas that are RENDERABLE objects — custom geometry types, custom lights. ✅ Correct — when your schema needs rendering support
The exam trap — Python-only schemas

Q25 offered "Create a Python-only schema by subclassing Usd.Schema." This is WRONG. Python-only schemas do not fully integrate with USD's type system. They cannot participate in C++ performance-critical code paths, don't support all USD features, and prim.IsA() may not work correctly. Full custom schemas require C++ code generation via usdGenSchema.

03 — The Complete Schema Creation Workflow

This is the exact workflow the exam tests across Q19, Q25, and Q26. Every step matters — missing one means your schema doesn't fully integrate.

1
Write schema.usda — the definition file
Define your schema class, what it inherits from, what attributes it has with their types and fallback values. This is the SOURCE of truth for your schema.
schema.usda
2
Run usdGenSchema — the code generator
This tool reads schema.usda and generates: C++ header files, C++ source files, Python bindings, and plugInfo.json for plugin registration. You do NOT write the C++ manually.
usdGenSchema schema.usda
3
Compile the generated C++ into a shared library
The generated C++ code inherits from UsdTyped (or UsdGeomImageable) and implements all the schema methods. You compile this into a .so / .dll plugin file.
cmake + make → mySchema.so
4
Register with the TfType system
The compiled plugin registers itself with TfType — USD's type identification system. This is what makes prim.IsA(MySchema) work. Without TfType registration, USD cannot find your schema type.
plugInfo.json → TfType registry
5
Deploy the plugin so USD can discover it
Place the plugin (.so/.dll + plugInfo.json) where USD can find it — either in PXR_PLUGINPATH_NAME or in the USD install directory. USD discovers plugins at startup.
PXR_PLUGINPATH_NAME=/path/to/myPlugin
6
Use your schema like any built-in USD schema
MySchema.Define(stage, "/path"). prim.IsA(MySchema). MySchema(prim).GetTemperatureAttr(). Schema fallbacks work. All USD tools respect the type.
from pxr import MySchemaLib
USDA + Pythonschema.usda definition and how the generated API looks
# ── schema.usda — the definition file ───────────────────────────────
#usda 1.0
(
    subLayers = [
        @usd/schema.usda@      # inherit from base USD schemas
    ]
)

over "GLOBAL" (
    customData = {
        string libraryName   = "acmeSensors"   # your library name
        string libraryPrefix = "AcmeSensor"    # prefix for C++ class names
    }
) {}

# ── Custom IsA schema — inherits from Xform (which is a UsdTyped) ───
class Xform "TemperatureSensor" (
    customData = {
        string className = "TemperatureSensor"
    }
    doc = """A prim representing a physical temperature sensor."""
    inherits = </Xform>   # Xform inherits from UsdTyped
)
{
    # Attributes with fallback values — these work without Set()
    float  sensor:temperature = 20.0
    float  sensor:minRange    = -40.0
    float  sensor:maxRange    = 125.0
    string sensor:serialNumber = ""
}

────────────────────────────────────────────

# ── After running usdGenSchema + compiling, this is how you use it ──
from pxr import Usd, AcmeSensors   # your generated module

stage = Usd.Stage.CreateInMemory()

# Define a prim using your custom IsA schema
sensor = AcmeSensors.TemperatureSensor.Define(stage, "/Factory/Sensor_001")

# Schema fallbacks work — even without Set()
print(sensor.GetTemperatureAttr().Get())   # 20.0 ← fallback value

# Type checking works properly
prim = sensor.GetPrim()
print(prim.GetTypeName())                       # "TemperatureSensor"
print(prim.IsA(AcmeSensors.TemperatureSensor)) # True
print(prim.IsA(UsdGeom.Xform))               # True — inherits from Xform

04 — The TfType System: What Registration Actually Means

TfType is USD's runtime type identification system. This is what makes prim.IsA(), prim.HasAPI(), and schema discovery work. The exam tests whether you understand WHY registration matters — not just that it exists.

Real-world analogy

TfType registration is like registering a business with the government. Your business (schema) can exist and operate privately without registration — but without it, nobody else can officially recognise it, it won't appear in any official directory, and government systems (USD tools, type checking) can't find or validate it. Registration makes it an official, discoverable, first-class citizen.

ConceptWhat TfType registration enables — and what breaks without it
# ── WITHOUT TfType registration ──────────────────────────────────────
#
# You might have a token "TemperatureSensor" in a USD file
# But USD has no idea what that means:
#
# prim.IsA(TemperatureSensor) → ERROR or always False
# prim.GetTypeName()          → "TemperatureSensor" (just a string)
# AcmeSensors.TemperatureSensor.Define() → fails to find the schema
# No fallback values             → Get() returns None not 20.0
# usdview shows unknown type     → no schema introspection
#
# ── WITH TfType registration ─────────────────────────────────────────
#
# USD knows this type exists, what it inherits from, what it contains:
#
# prim.IsA(TemperatureSensor) → True (type lookup works)
# prim.IsA(UsdGeomXform)      → True (inheritance chain known)
# Fallback values             → 20.0 returned from schema definition
# usdview shows full schema   → all attributes with types and values
# Schema API methods work     → GetTemperatureAttr() etc.
#
# ── HOW REGISTRATION HAPPENS ────────────────────────────────────────
#
# usdGenSchema creates plugInfo.json
# plugInfo.json declares the plugin and its types to the TfType system
# When USD loads, it reads all plugInfo.json files in plugin paths
# Each type described therein gets registered in the TfType registry
# Now IsA(), HasAPI() can look up and resolve those types
#
# KEY EXAM POINT:
# "Registering the token in TfType system WITHOUT any schema or API linkage"
# → This is INSUFFICIENT. The type needs full schema linkage.
# Token alone does not provide inheritance chain, fallbacks, or API.
The exact exam trap — Q27

"Registering the custom model kind token in the TfType system without any schema or API linkage" — this was marked INCORRECT. Token registration alone is insufficient. The token needs to be linked with UsdModelAPI and proper schema definitions to function correctly. Think of the token as just a name — without the schema behind it, the name means nothing to USD.

05 — File Format Plugins: SdfFileFormat

Q26 tested whether you know how to add support for a new file format (like .abc, .fbx, .myformat) to USD. This is completely separate from custom schemas — schemas define scene data types, file format plugins define how data is READ and WRITTEN.

Real-world analogy

A custom schema is like defining a new type of document (a "sensor report"). A file format plugin is like teaching Microsoft Word to open PDF files. These are separate concerns. The schema defines what the data means. The file format plugin defines how to load and save a specific file format. USD already has schema types — you might just need to teach USD to read your company's proprietary .mysim format.

Concept + C++ structureSdfFileFormat interface — the complete picture
# ── WHAT SdfFileFormat IS ────────────────────────────────────────────
#
# SdfFileFormat is the C++ interface that defines:
#   - How USD reads a file of your format into a SdfLayer
#   - How USD writes a SdfLayer back to your format
#   - What file extensions your format uses (.mysim, .abc etc.)
#   - The format's name and version
#
# ── THE CORRECT IMPLEMENTATION STEPS ───────────────────────────────
#
# 1. Create a C++ class inheriting from SdfFileFormat
#    class MyFormatPlugin : public SdfFileFormat { ... }
#
# 2. Implement the required methods:
#    bool Read(SdfLayer*, const string& resolvedPath, bool metadataOnly)
#    bool WriteToFile(const SdfLayer&, const string& filePath, ...)
#
# 3. Register with TfType so USD discovers the plugin:
#    TF_REGISTRY_FUNCTION(TfType) {
#        TfType::Define<MyFormatPlugin, TfType::Bases<SdfFileFormat>>();
#    }
#
# 4. Declare in plugInfo.json so USD finds it at startup
#
# ── WHAT HAPPENS WHEN USD OPENS A .mysim FILE ───────────────────────
#
# stage = Usd.Stage.Open("scene.mysim")
# USD sees extension .mysim
# Looks up registered SdfFileFormat plugins
# Finds MyFormatPlugin (registered for .mysim)
# Calls MyFormatPlugin.Read() to load the data into a SdfLayer
# Stage composes normally from that point
#
# ── WRONG APPROACHES (exam traps) ───────────────────────────────────
#
# ❌ Override UsdStage::Open()
#    UsdStage::Open is a static method, not designed to be overridden
#    File loading happens at the SdfLayer level, not the Stage level
#
# ❌ Modify USD core source code
#    USD is designed for extensibility via plugins
#    Modifying core breaks compatibility with other USD tools
#    Plugin mechanism exists specifically to avoid this
#
# ❌ Create a new UsdSchema for the file format
#    Schemas define scene DESCRIPTION TYPES (what prims are)
#    They do NOT control file I/O or parsing
#    These are completely separate concerns
The key separation to memorise

UsdSchema = defines what scene data IS (prim types, attributes, relationships). SdfFileFormat = defines how data is READ and WRITTEN from/to files. Custom schema for new prim types. SdfFileFormat plugin for new file formats. They are independent — you can have either without the other.

06 — Custom Model Kinds + UsdModelKindRegistry

Q24 and Q27 both tested custom model kinds. The key thing the exam tests is the difference between the STANDARD kind system (assembly/group/component/subcomponent) and EXTENDING it with custom kinds.

Real-world analogy

The standard model kinds (assembly, group, component, subcomponent) are like the standard job titles at a company (CEO, Manager, Employee, Intern). Custom model kinds are like adding new roles specific to your company (Factory_Unit, Sensor_Array, ProductionLine). You don't change the existing titles — you register new ones in the HR system (UsdModelKindRegistry) so the whole company (USD ecosystem) recognises them.

Python + ConceptThe complete custom model kind system
# ── THE STANDARD KIND HIERARCHY (built into USD) ─────────────────────
#
# assembly     → top-level published collection
# group        → organisational container
# component    → leaf reusable asset (most common for individual assets)
# subcomponent → important internal node within a component
#
# ── ADDING CUSTOM KINDS ─────────────────────────────────────────────
#
# Step 1: Define a unique token for your new kind
#   "factory_unit" — a kind for factory floor assets
#
# Step 2: Register it with UsdModelKindRegistry
#   This is the REGISTRY for model kinds (different from TfType)
#   Without registration USD doesn't know factory_unit is a valid kind
#   It also needs to know the parent kind in the hierarchy
#   (e.g. factory_unit is a subkind of component)
#
from pxr import Usd, Kind

# Step 3: Apply the custom kind to prims using UsdModelAPI
stage = Usd.Stage.CreateInMemory()
prim = stage.DefinePrim("/Factory/Unit_A", "Xform")

model_api = Usd.ModelAPI(prim)
model_api.SetKind("factory_unit")
# This sets:  kind = "factory_unit"  in the prim's metadata

print(model_api.GetKind())   # "factory_unit"

# Step 4: Implement custom schema plugin inheriting from UsdModelAPI
#   This encapsulates specialised behaviours and validation for factory_unit
#   EXTEND UsdModelAPI — do NOT override it
#   Your plugin ADDS to UsdModelAPI's capabilities
#
# Step 5: Extend UsdModelAPI's Validate() method
#   Add custom validation rules specific to factory_unit:
#   "factory_unit prims must have a sensor:serialNumber attribute"
#   "factory_unit prims must have at least one child sensor"
#   This ensures structural integrity
#
# ── EXAM TRAPS (what NOT to do) ──────────────────────────────────────
#
# ❌ Override UsdModelAPI — this replaces existing behaviour, breaks things
#    You must EXTEND (add to), not OVERRIDE (replace)
#
# ❌ Register token in TfType WITHOUT schema linkage
#    Token alone gives you a name but no behaviour, validation, or API
#
# ❌ Use variants WITHIN the model kind to handle different behaviours
#    Variants are for content variation (LOD, colour)
#    Model kind is about CLASSIFICATION, not content variation
#    These are completely separate concepts
#
# ❌ Modify USD core source
#    USD's plugin and registry system exists precisely to avoid this
Extend vs Override — the critical distinction Q27 tested

The exam marked "Overriding the default UsdModelAPI schema to change the behavior of existing model kinds" as WRONG. The word "overriding" means replacing existing behaviour — which breaks all existing prims using the standard kinds. "Implementing a custom schema plugin that INHERITS FROM UsdModelAPI" is CORRECT — you ADD new capabilities for your custom kind without touching the existing ones. Extend = add to. Override = replace. Always extend in USD.

07 — Variant Fallback Selections

Q22 tested variant fallback selections — a topic not covered in the Day 3/4 variant notes. This is an advanced variant topic about what happens when a variant selection doesn't exist.

Real-world analogy

Imagine a restaurant menu with language variants — English, French, Spanish. A customer who speaks Portuguese opens the menu. There's no Portuguese option. The fallback selection tells the system: "If Portuguese is requested but doesn't exist, try Spanish first, then English." Without a fallback, the customer gets an empty menu. With a fallback, they get the best available alternative.

The problem variant fallbacks solve

In production, not every variant selection exists in every asset. A character might have "lod_high", "lod_medium", "lod_low" variants. In a render farm environment, someone accidentally requests "lod_ultra" which doesn't exist. Without fallbacks, the variant fails. With fallbacks, USD tries "lod_high" instead.

PythonAll three correct variant fallback approaches
from pxr import Usd

# ── APPROACH 1: LayerStack fallback arcs ─────────────────────────────
# Specify fallback variant selections in the stage's metadata
# These are variant fallback arcs — explicit fallback definitions
# in the layer stack
#
# In USDA at the root of a layer:
# (
#     variantFallbacks = {
#         "lod":   ["high", "medium", "low"]
#         "color": ["default"]
#     }
# )
# If "lod_ultra" requested: try "high" first, then "medium", then "low"

# ── APPROACH 2: UsdVariantFallback API — programmatic runtime control ─
# The UsdVariantFallback API lets you SET fallbacks programmatically
# at runtime — key for dynamic or environment-specific resolution

fallbacks = {
    "lod":   ["high", "medium", "low"],
    "color": ["default"]
}
# Set global fallbacks — applies to all stages opened after this
Usd.Stage.SetGlobalVariantFallbacks(fallbacks)

# Now when any stage opens with an unknown lod variant,
# it tries "high" first, then "medium", then "low"
# This is the UsdVariantFallback API for programmatic control

stage = Usd.Stage.Open("scene.usda")
# Stage uses the fallbacks set above

# ── APPROACH 3: Session layer for user-specific fallbacks ────────────
# Define fallback variants in the SESSION layer
# This lets individual users customise their fallback preferences
# WITHOUT modifying the shared base USD files
# Session layer = strongest + in-memory + per-session
# Perfect for user-specific or environment-specific overrides

session = stage.GetSessionLayer()
stage.SetEditTarget(session)
# Write fallback variant selections into session
# They only affect this session, never saved to files

# ── WRONG APPROACHES ────────────────────────────────────────────────
#
# ❌ Embed fallback selections in prim metadata
#    Prim metadata is not designed for fallback logic
#    This leads to unpredictable behaviour
#    Fallbacks belong at the stage/layer level, not the prim level
#
# ❌ Rely only on default variant selection in the schema
#    Default selections alone don't cover cases where the
#    requested variant doesn't exist at all
#    Fallback lists are needed for robustness
#
# ❌ Use fallback selections only during composition, not runtime
#    Fallbacks ARE relevant at runtime too — dynamic scenes
#    can request variants that need fallback resolution

08 — Exam Pattern Recognition

The custom schema questions on the real exam follow very specific patterns. Once you recognise the pattern in the question, you can eliminate wrong answers systematically.

Pattern 1 — "How do you create a custom schema?"

Answer framework

1. Write schema.usda. 2. Inherit from UsdTyped or UsdGeomImageable (NOT UsdSchemaBase directly). 3. Run usdGenSchema to generate C++ and Python. 4. Register with TfType system. 5. Deploy as plugin. Never: modify core, Python-only, inherit directly from UsdSchemaBase.

Pattern 2 — "How do you add a new file format?"

Answer framework

Implement the SdfFileFormat interface + register with TfType system. Never: override UsdStage::Open(), modify core source, create a UsdSchema for it.

Pattern 3 — "How do you create custom model kinds?"

Answer framework

Define unique token + register with UsdModelKindRegistry + set 'kind' metadata via UsdModelAPI + implement schema plugin that inherits (not overrides) UsdModelAPI + extend Validate() method. Never: use variants for this, register token alone, override UsdModelAPI.

Pattern 4 — "Extend vs Override"

Critical exam distinction

Any question option saying "override" an existing schema, API, or core system = WRONG. USD is always extended, never overridden. "Override" breaks existing functionality for all users of that schema. "Extend" (inherit from, add to) maintains backward compatibility while adding new capabilities. When you see "overriding" in an option about customising USD, eliminate it immediately.

Quick elimination checklist for custom schema questions

Exam strategyImmediate eliminations when you see these phrases
# IMMEDIATELY ELIMINATE any option containing:
#
# "Modify the core USD source code"
# → Always wrong. USD uses plugins, not core modifications.
#
# "Python-only schema" or "subclassing Usd.Schema in Python"
# → Always wrong. Full schemas need C++ + usdGenSchema.
#
# "Inheriting directly from UsdSchemaBase"
# → Wrong for typical use. Correct base is UsdTyped or UsdGeomImageable.
#
# "Overriding" any existing schema, API, or method
# → Wrong. Always EXTEND (inherit from, add to).
#
# "Override UsdStage::Open()"
# → Wrong. File formats use SdfFileFormat, not Stage::Open override.
#
# "Register token in TfType WITHOUT schema or API linkage"
# → Wrong. Token alone is insufficient without full schema backing.
#
# "Define schema in a .usd file at runtime"
# → Wrong. Schemas are defined in schema.usda + compiled, not .usd files.
#
# "Create new UsdSchema to control file format I/O"
# → Wrong. Schemas ≠ file I/O. Use SdfFileFormat for file format plugins.
#
# "Embed fallback variant selections in prim metadata"
# → Wrong. Fallbacks belong at stage/layer level via UsdVariantFallback API.

Custom schemas — the 12 things to know cold

1Custom schemas use: schema.usda → usdGenSchema → C++ compile → TfType register → plugin deploy. This is the complete chain.
2Inherit from UsdTyped (standard) or UsdGeomImageable (renderable). NOT directly from UsdSchemaBase. NOT Python-only.
3TfType registration is what makes IsA(), HasAPI(), fallback values, and schema discovery work. Token registration alone is insufficient.
4File format plugins use the SdfFileFormat interface — completely separate from schema types. Never override UsdStage::Open().
5Custom model kinds: unique token + UsdModelKindRegistry + 'kind' metadata on root prim + schema plugin inheriting UsdModelAPI + extended Validate().
6Always EXTEND UsdModelAPI — never override it. Overriding replaces behaviour. Extending adds new capabilities.
7Variant fallbacks: UsdStage.SetGlobalVariantFallbacks() for programmatic control, LayerStack variantFallbacks for explicit arcs, session layer for user-specific preferences.
8Variant fallbacks are NOT embedded in prim metadata. They live at stage/layer level. Default selections alone are insufficient for missing variants.
9Variants within model kinds = WRONG. Variants serve content variation. Model kinds serve classification. These are separate systems.
10Never modify core USD source code. USD's entire extensibility architecture (plugins, schemas, file formats) exists to avoid this.
11SdfFileFormat = file I/O. UsdSchema = scene data types. Custom schema for new prim types. SdfFileFormat for new file formats. Independent concerns.
12Elimination rule: if an option says "override", "modify core", "Python-only", or "token without linkage" — eliminate it immediately.
