01 — Variant Sets

A Variant Set is a named group of switchable options on a prim. Think of it as a dropdown menu baked into your asset — you define the choices, and whoever references your asset can pick which one they want without modifying your original file.

Real-world analogy

Variants are like a car configurator on a website. One car model, multiple choices: colour (red / blue / silver), trim level (standard / sport / luxury), roof (solid / sunroof). Each option is a variant. The car itself is one asset. You don't have 12 separate car files — you have one file with three variant sets that combine into any configuration.

Variants are the V in LIVERPS — the third strongest composition arc. They are stronger than References and Payloads but weaker than Local opinions and Inherits.

A single asset with two variant sets — colour and LOD

USDA syntax — what variant sets look like in a file

USDAA cube with a colour variant set
#usda 1.0
(
    defaultPrim = "Cube"
)

def Cube "Cube" (
    # ── DECLARE the variant set exists on this prim ───────────────────
    prepend variantSets = "color"
    # This tells USD: "this prim has a variant set named 'color'"

    # ── SELECT which variant is active ────────────────────────────────
    variants = {
        string color = "red"
        # "red" is the CURRENTLY SELECTED variant
        # This can be overridden in a stronger layer (e.g. from scene.usda)
    }
)
{
    # ── DEFINE what each variant contains ─────────────────────────────
    variantSet "color" = {
        "red" {
            # opinions inside here ONLY apply when color = "red"
            color3f[] primvars:displayColor = [(1, 0, 0)]
        }
        "blue" {
            # opinions inside here ONLY apply when color = "blue"
            color3f[] primvars:displayColor = [(0, 0, 1)]
        }
        "green" {
            color3f[] primvars:displayColor = [(0, 1, 0)]
        }
    }
}
Key pattern — metadata block vs data block

Notice: prepend variantSets and variants = {} live in the ( ) metadata block. The actual variantSet "color" = { } definition lives in the { } data block. This split is intentional and appears in every exam question about variants.

Switching variants from another layer

One of the most powerful things about variants is that a referencing scene can pick a different variant without touching the asset file. This is the V in LIVERPS — the variant selection in a stronger layer overrides the default selection in the asset.

USDAscene.usda overrides the variant selection from outside the asset
# scene.usda — the referencing scene
def Xform "World"
{
    def Xform "MyCube" (
        prepend references = @./cube.usda@
        # cube.usda defaults to color = "red"
        # BUT we can override the variant selection here:
        variants = {
            string color = "blue"
            # This LOCAL variant selection is STRONGER than the
            # default "red" selection inside cube.usda
            # Result: cube renders blue in this scene
        }
    )
    { }
}

Python API for variant sets

PythonCreating and switching variants in code
from pxr import Usd, UsdGeom

stage = Usd.Stage.CreateNew("car.usda")
car = UsdGeom.Xform.Define(stage, "/RaceCar")
prim = car.GetPrim()

# ── CREATE A VARIANT SET ─────────────────────────────────────────────
vsets = prim.GetVariantSets()
# GetVariantSets() returns the UsdVariantSets manager for this prim

color_vset = vsets.AddVariantSet("color")
# Creates the "color" variant set on the prim (or gets it if it exists)

# ── ADD VARIANTS AND AUTHOR DATA INSIDE THEM ────────────────────────
color_vset.AddVariant("red")
color_vset.SetVariantSelection("red")   # select it so we can edit it

with color_vset.GetVariantEditContext():
    # EVERYTHING inside this 'with' block is authored INSIDE the "red" variant
    # When color = "red" is selected, these opinions are composed in
    # When a different variant is selected, these opinions are NOT composed
    cube = UsdGeom.Cube.Define(stage, "/RaceCar/Body")
    cube.GetDisplayColorAttr().Set[(1.0, 0.0, 0.0)]

color_vset.AddVariant("blue")
color_vset.SetVariantSelection("blue")

with color_vset.GetVariantEditContext():
    # Authored inside the "blue" variant
    cube = UsdGeom.Cube.Define(stage, "/RaceCar/Body")
    cube.GetDisplayColorAttr().Set([(0.0, 0.0, 1.0)])

# ── SET THE DEFAULT VARIANT SELECTION ───────────────────────────────
color_vset.SetVariantSelection("red")
# Sets "red" as the default — what you see when you open the stage

# ── READ THE CURRENT SELECTION ──────────────────────────────────────
current = color_vset.GetVariantSelection()
print(current)   # "red"

# ── GET ALL AVAILABLE VARIANT NAMES ─────────────────────────────────
names = color_vset.GetVariantNames()
print(names)   # ["red", "blue"]

# ── SWITCH VARIANT FROM OUTSIDE (what a referencing scene does) ─────
color_vset.SetVariantSelection("blue")
# If called from a stronger layer, this overrides the default

stage.Save()
Fast vs slow variant operations

Some variant operations are fast — changing an attribute value or visibility (USD only needs to re-resolve values, not rebuild the scene structure). Others are slow — swapping a reference path or changing activation (USD must rebuild parts of its composition index). Design variant sets with this in mind: use attribute overrides for frequently toggled options, reserve structural changes for rare switches.

02 — When NOT to Use Variants

This is directly tested in the exam as topic 1.7: "Identify situations where variants are/are not appropriate for structuring assets." Knowing when NOT to use variants is just as important as knowing how to use them.

⚠️
The core risk: Variant combinations multiply. Two variant sets with 4 options each = 16 combinations. Three variant sets with 4 options each = 64. You have to author data for every combination you want to support — and some combinations may not make logical sense. Getting carried away creates unmaintainable assets.
Situation Use Variants? Why
A chair that comes in red, blue, green versions ✅ Yes Discrete, finite, switchable options — perfect for variants
A mesh at three levels of detail (hero/medium/proxy) ✅ Yes LOD is a classic variant use case — one asset, three geometric representations
A paint colour that could be any RGB value ❌ No Continuous values can't be discretised into variants. Use a primvar instead — it's infinitely flexible
50 different building facade textures ❌ No Too many variants = bloated asset. Use references to swap materials instead
Combinations that don't make logical sense ❌ No e.g. "wet character without hat" — you'd have to author data for an impossible combination
Options only needed for one specific shot ❌ No Bake the override as a local opinion in that shot's layer instead — don't pollute the asset
Controlling which of several independent sub-assets appears ✅ Yes Variants can activate/deactivate prims or swap references — good for modular assembly

Use variants when

  • Options are discrete and finite
  • The same asset is used in many scenes with different configurations
  • You want a clean UI dropdown in USD tools
  • Switching is non-destructive (no file edits needed)

Use primvars instead when

  • Values are continuous (any colour, any number)
  • The variation needs to be per-instance (different for each copy)
  • The number of options would be very large
  • You need to drive the value from a shader

03 — Inherits Arc

The Inherits arc lets one prim broadcast its opinions to all other prims that declare they inherit from it. Change the source prim, and every inheriting prim automatically picks up that change — across the entire scene, even across multiple referenced files.

Real-world analogy

Think of inherits like a company-wide email announcement. You send one email to the whole company (the class prim). Everyone who reads it (the inheriting prims) now knows the new policy. If you send a follow-up correction, everyone automatically gets the update. But if an individual employee has their own personal note that contradicts the announcement (a local opinion), their personal note wins — they ignore the broadcast.

Inherits is the I in LIVERPS — the second strongest arc, beaten only by Local opinions. This means you can still override anything inherited by just writing a local opinion in a stronger layer.

Inherits — one class prim broadcasts to many prims across the scene

USDA syntax for inherits

USDADefining a class and inheriting from it
#usda 1.0

# ── 1. DEFINE THE CLASS PRIM (template — never rendered) ─────────────
class Xform "_PineTree"
# 'class' specifier = template, skipped by renderers and default traversals
# The underscore prefix is a CONVENTION, not required, but signals "abstract"
{
    double height = 5.0
    color3f[] primvars:displayColor = [(0.1, 0.5, 0.1)]
}

# ── 2. CREATE PRIMS THAT INHERIT FROM THE CLASS ───────────────────────
def Xform "Forest"
{
    def Xform "PineTree_A" (
        prepend inherits = </_PineTree>
        # inherits uses a PRIM PATH (in <>), NOT a file path (in @@)
        # This means the class prim must exist in THIS stage's namespace
    )
    {
        # No local overrides — gets everything from _PineTree
        # height = 5.0, displayColor = green
    }

    def Xform "PineTree_C" (
        prepend inherits = </_PineTree>
    )
    {
        double height = 8.0
        # LOCAL opinion on height — STRONGER than the inherited value
        # This tree is taller. Still inherits the displayColor from _PineTree.
    }
}
Key difference — inherits uses paths, not files

References use file paths: @./chair.usda@. Inherits uses prim paths: </_PineTree>. The class prim must exist somewhere in the composed stage's namespace. You cannot inherit from a prim in a completely separate stage.

PythonAdding an inherits arc in code
from pxr import Usd, UsdGeom, Sdf

stage = Usd.Stage.CreateNew("forest.usda")

# ── CREATE THE CLASS PRIM ─────────────────────────────────────────────
class_prim = stage.CreateClassPrim("/_PineTree")
# CreateClassPrim() creates a prim with the 'class' specifier
# Note: path starts with / even though it's a class

# Add some data to the class
xf = UsdGeom.Xform(class_prim)
height_attr = class_prim.CreateAttribute("height", Sdf.ValueTypeNames.Double)
height_attr.Set(5.0)

# ── CREATE A PRIM THAT INHERITS FROM IT ─────────────────────────────
tree_a = UsdGeom.Xform.Define(stage, "/Forest/PineTree_A")

inherits = tree_a.GetPrim().GetInherits()
# GetInherits() returns the UsdInherits manager for this prim

inherits.AddInherit(Sdf.Path("/_PineTree"))
# Add the inherit arc pointing at the class prim's path
# Now PineTree_A gets height=5.0 from _PineTree

stage.Save()

04 — Specializes Arc

Specializes is the weakest composition arc in LIVERPS — the S at the end. It works similarly to Inherits but with a crucial difference: instead of broadcasting strong opinions, it provides fallback values. Any other opinion from any other arc will override a specializes value.

Real-world analogy

Specializes is like a default settings file for an app. The app ships with defaults (the base class). Users can override any setting they want and their setting wins. But for anything they haven't touched, the app falls back to the default. The default is the weakest voice — it only speaks when nobody else does.

The classic use case is material libraries. You have a base Plastic material. RoughPlastic specializes from it, reducing the glossiness. But any subsequent opinion on RoughPlastic (from a variant, a reference, anything) will override the specialized value — because specializes is the weakest arc.

USDASpecializes for material inheritance chains
# materials.usda
class "_BasePlastic"
{
    float inputs:roughness = 0.2   # smooth plastic default
    float inputs:metallic  = 0.0   # not metallic
    color3f inputs:diffuseColor = (0.8, 0.8, 0.8)  # grey
}

def Material "RoughPlastic" (
    prepend specializes = </_BasePlastic>
    # specializes = "I am a specialisation of _BasePlastic"
    # I get its values as FALLBACKS — weaker than everything else
)
{
    float inputs:roughness = 0.8
    # This local opinion on roughness OVERRIDES the specializes value (0.2)
    # Because local (L) beats specializes (S) in LIVERPS

    # metallic and diffuseColor are NOT locally set
    # → they fall back to _BasePlastic values (0.0 and grey)
}

# If someone later changes _BasePlastic.diffuseColor to blue,
# RoughPlastic automatically becomes blue too — it inherits the fallback.
# But if RoughPlastic had its own local diffuseColor, that would win.

05 — Inherits vs Specializes

These two arcs look similar but behave very differently. This is one of the most commonly tested distinctions in the exam. Here's the clearest way to understand the difference.

I
Inherits

Strong broadcast. The class prim's opinions are STRONGER than references and payloads. Used to enforce shared properties that should be hard to override — like a base look that all instances must share.

V
Variants

Switchable options. Stronger than references. Used for discrete choices (colour, LOD) that the referencing scene can select.

S
Specializes

Weak fallback. The weakest arc of all. Used to define defaults that anything else can override — like base material properties for a material library.

The key difference — inherits pushes opinions UP, specializes provides fallbacks DOWN
Property Inherits (I) Specializes (S)
LIVERPS strength2nd — very strongLast — weakest of all
What it providesStrong opinions that are hard to overrideFallback defaults that anything overrides
Change the source prim — who updates?All inheritors update (unless they have a local override)All specializers update (unless they have any other opinion)
Typical use caseShared base properties for a group of similar assetsMaterial library base classes, default parameter sets
Source prim specifierclassclass
USDA syntaxprepend inherits = </Path>prepend specializes = </Path>
The one-line distinction to memorise

Inherits = "I want to share these properties and make them hard to override." Specializes = "I want to provide fallback defaults that anything else can easily override." When in doubt: Inherits is strong, Specializes is weak.

06 — Multi-User Scene Layering

Exam topic 1.5 asks you to "design potential scene layering strategies to facilitate multi-user workflows." This is about how to structure sublayers so multiple artists can work on the same shot without stepping on each other.

Real-world analogy

Think of it like a shared Google Doc where each person has their own colour of text. Everyone edits the same document simultaneously, but you can see exactly who wrote what. In USD: each artist works in their own layer, all layers are sublayered together, and the strongest layer's opinion wins when there's a conflict.

A typical shot pipeline — each discipline in its own layer

Each artist works in their own file. They use over to modify prims they don't own. The shot file just lists them in order. Changing the order changes who wins on any conflicting property.

Splitting a monolithic asset for collaboration (exam topic 1.11)

When an asset starts as one big file and needs to be split across multiple workstreams:

USDASplitting a monolithic asset into collaborative workstreams
# BEFORE: one monolithic file that everyone edits (causes conflicts)
# robot.usda — everyone edits this, conflicts everywhere

────────────────────────────────────────────

# AFTER: split into layers by discipline

# robot.usda — the root layer (nobody edits this directly)
#usda 1.0
(
    subLayers = [
        @./robot_anim.usda@,      # animator owns this
        @./robot_rigging.usda@,    # rigger owns this
        @./robot_shading.usda@,    # shading artist owns this
        @./robot_geo.usda@         # modeller owns this (weakest = base)
    ]
)

# robot_geo.usda (modeller's file)
def Xform "Robot" {          # modeller DEFINES the prim (they own it)
    def Mesh "Body" { }
    def Mesh "Arm_L" { }
}

# robot_anim.usda (animator's file)
over "Robot" {               # animator OVERRIDES (doesn't own)
    over "Arm_L" {
        double3 xformOp:translate.timeSamples = {
            0: (0,0,0), 24: (1,2,0)
        }
    }
}

07 — Encapsulation

Encapsulation is one of the most important and most misunderstood concepts in USD composition. It explains a whole category of "why isn't my opinion taking effect?" bugs.

The rule is: opinions from inside a referenced asset's own layer stack cannot be overridden from outside that layer stack using the same composition arc. The referenced asset's local layer stack is self-contained — encapsulated.

Real-world analogy

Encapsulation is like ordering a sealed factory product. The factory shipped you a chair with its own internal parts (the chair's sublayers). You can paint the chair a different colour (local override from outside). But you can't change how the factory assembled the internal parts without opening the factory's own files. The factory's internal work is encapsulated — you interact only with the interface it exposes.

USDAEncapsulation — why a sublayer inside a reference cannot be overridden from outside
# chair.usda — the asset file
#usda 1.0
(
    defaultPrim = "Chair"
    subLayers = [@./chair_anim.usda@]
    # chair_anim.usda is INSIDE chair.usda's layer stack
    # It's encapsulated — part of the chair's internal composition
)
def Xform "Chair" {
    double3 xformOp:translate = (0, 0, 0)
}

────────────────────────────────────────────

# scene.usda — the referencing scene
def Xform "MyChair" (
    prepend references = @./chair.usda@
)
{
    double3 xformOp:translate = (5, 0, 0)
    # ✅ This LOCAL opinion OVERRIDES chair.usda's translate.
    # Local beats Reference in LIVERPS. This works fine.
}

# BUT: you CANNOT override what's inside chair_anim.usda
# from scene.usda using another sublayer of scene.usda.
# chair_anim.usda is encapsulated inside chair.usda's layer stack.
# Its opinions are "inside the reference boundary" — you only
# get to override the composed result from outside.
Exam trap — this is why opinions sometimes don't take effect

If you sublayer an override into a scene and it doesn't affect something inside a referenced asset, encapsulation is likely the reason. The referenced asset's internal sublayers are self-contained. You can only override the referenced prim's final composed values from outside — using local opinions on the referencing prim, or by editing the asset file itself.

08 — Q13 Walkthrough: Variant Debugging

Question 13 from the study guide is the hardest debugging question. It involves THREE files, a variant set, and a payload. Let's trace it exactly.

The scenario

An artist sets loftedColor = "blue" in main.usda. They expect the cube to turn blue. It doesn't. Why?

USDAmain.usda — the root layer
def Xform "World"
{
    def Xform "MyBox" (
        prepend references = @./MyBox.usda@
        variants = {
            string loftedColor = "blue"
            # Artist sets this to "blue" hoping the cube turns blue
        }
    )
    {
        over "Cube" (
            variants = {
                string color = "red"
                # A direct LOCAL opinion setting the nested color variant to "red"
            }
        )
        { }
    }
}
USDAMyBox.usda — the referenced asset
def Xform "MyBox" (
    prepend payload = @./contents.usda@
    prepend variantSets = "loftedColor"
    variants = {
        string loftedColor = "red"   # default selection
    }
)
{
    variantSet "loftedColor" = {
        "red" {
            over "Cube" (
                variants = { string color = "red" }
            ) { }
        }
        "blue" {
            over "Cube" (
                variants = { string color = "blue" }
                # When loftedColor="blue", this sets Cube's color variant to "blue"
            ) { }
        }
    }
}
USDAcontents.usda — the payload (the actual geometry)
def Xform "MyBox"
{
    def Cube "Cube" (
        prepend variantSets = "color"
        variants = { string color = "red" }
    )
    {
        variantSet "color" = {
            "red"  { color3f[] primvars:displayColor = [(1,0,0)] }
            "blue" { color3f[] primvars:displayColor = [(0,0,1)] }
        }
    }
}

Tracing the bug — step by step

The chain is: main.usda sets loftedColor = "blue" → that should set Cube.color = "blue" via MyBox.usda's loftedColor variant → that should compose in displayColor = (0,0,1) from contents.usda.

But here's what actually happens:

The bug — a direct local opinion is stronger than the variant

Inside main.usda, there is a direct over "Cube" block with variants = {string color = "red"}. This is a local opinion (L in LIVERPS) on the Cube's color variant selection. Local opinions beat Variants (V) in LIVERPS. So no matter what loftedColor is set to, the local color = "red" opinion ALWAYS wins — it overrides whatever the loftedColor variant tries to set.

The correct answer to Q13 is B: "The direct local opinion for the color variant selection in main.usda is stronger than the opinion of the color variant selection authored by loftedColor."

Why the box stays red — LIVERPS applied to the color variant selection
The fix

Remove the direct over "Cube" (variants = {string color = "red"}) from main.usda. Once that local opinion is gone, the loftedColor variant is free to set the Cube's color variant selection — and setting loftedColor to "blue" will correctly make the cube turn blue.

Day 3 key takeaways — commit these to memory

1A Variant Set is a named group of switchable options on a prim. Define it in the asset, select it from outside. The V in LIVERPS — stronger than references, weaker than local and inherits.
2Variants are good for discrete, finite options (colour, LOD, on/off). Avoid variants for continuous values, too many options, or logically impossible combinations — use primvars instead.
3Variant data lives inside a variantSet "name" = { } block in { } curly braces. The selection lives in variants = { string name = "choice" } in ( ) metadata.
4Inherits (I) is a strong broadcast arc — the class prim's opinions propagate to all inheritors. Change the class, all inheritors update. Good for enforcing shared base properties.
5Specializes (S) is the weakest arc — provides fallback defaults that ANY other opinion overrides. Used for material library base classes. Good for "inherit this unless something else says otherwise."
6Both Inherits and Specializes target class prims using prim paths (in <>), not file paths. The class prim must exist in the composed stage namespace.
7Multi-user workflows: each discipline gets their own sublayer. Modellers use def (they own prims). Everyone else uses over. The root layer just lists them in strength order.
8Encapsulation: you cannot override a referenced asset's internal sublayer opinions from outside the reference boundary. You can only override the composed result via local opinions on the referencing prim.
9Q13 bug: a direct local opinion on a variant selection beats what a variant set tries to set — because L beats V in LIVERPS. The loftedColor variant couldn't change the color because main.usda had a stronger local opinion locking it to "red".
10Full LIVERPS in order: Local → Inherits → Variants → rElocates → References → Payloads → Specializes. You now know all six arcs.
