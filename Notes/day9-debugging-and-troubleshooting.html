01 — Debugging Mindset

Almost every USD debugging scenario comes down to one of five root causes. Identifying the type immediately tells you which tool to reach for.

Root cause 1 — Wrong value winning

A property shows the wrong value. Some layer has a stronger opinion than expected. Tool: PropertyStack, GetPropertyStack()

Root cause 2 — Wrong structure / missing prims

Prims are missing despite being in source layers. Usually invalid references, namespace/path mismatches, or broken composition arcs. Tool: PrimStack, usdview composition tab, usdchecker

Root cause 3 — Wrong layer getting edits

Value authored but went to wrong layer. Silent, no error. Tool: Check GetEditTarget() before authoring

Root cause 4 — Composition arc conflict

Unexpected overrides from references, variants, or inherits. Tool: usdview composition debugger, Usd.Debug.PrintComposition, usdcat --print-composition

Root cause 5 — Performance / loading

Stage loads slowly or viewport is laggy. Tool: TfDebug, check payload loading, verify instancing

Root cause 6 — Schema version mismatch

Unexpected behavior from attribute differences across layers using different schema versions. Tool: Validate schema versions across all layers

The most reliable symptom of poor authoring — exam Q49

The most reliable indicator of poorly authored USD data (invalid references or incorrect namespace usage) is: unexpected prims missing from the stage despite being present in source layers. This directly reflects broken composition arcs from invalid references or namespace collisions. Slow rendering, API deprecation warnings, and successful payload loading are NOT indicators of poor authoring.

03 — usdview Debugging Panels

The exam tests usdview panels specifically. Know each panel, what it shows, and when to use it.

The Composition Tab — most important for the exam

The composition debugger (Composition tab in Metadata/Composition panel) shows the full breakdown of how a prim is composed — every arc, every layer, every contribution. This is the first tool to reach for when a prim has unexpected values or a composition arc is not resolving correctly.

usdview workflowUsing the composition debugger step by step
# PROBLEM: /World/Chair has translate = (99,0,0) — expected (5,0,0)
#
# STEP 1: Open usdview
#   .\\scripts\\usdview.bat scene.usda
#
# STEP 2: Click /World/Chair in the Scenegraph panel
#
# STEP 3: Open the Metadata/Composition panel → Composition Tab
#   This shows the FULL composition arc graph for this prim
#   Every reference, sublayer, variant, inherit is listed
#   Each arc shows which layer it came from
#
# STEP 4: Open the LayerStack Tab
#   Shows all layers contributing to this prim, strongest → weakest
#   Shows which value each layer has for translate
#   The FIRST layer with an opinion = the one that wins
#
# STEP 5: Open the Layers panel (View → Layers)
#   Lists ALL layers in the stage
#   You can TOGGLE layers on/off interactively
#   Toggle a layer off → see if the wrong value disappears
#   That tells you which layer is responsible
usdview composition debugger — what it specifically provides

The composition debugger gives a visual, detailed breakdown of how layers and references compose for a selected prim. It helps pinpoint conflicting opinions and unexpected overrides. This is the exam's preferred "first tool" for unexpected composition results. It is NOT just a scene browser — it exposes the full arc-level composition structure.

04 — PrimStack, PropertyStack + PrintComposition
PythonAll three composition inspection APIs
from pxr import Usd, UsdUtils

stage = Usd.Stage.Open("scene.usda")
prim  = stage.GetPrimAtPath("/World/Chair")

# ── 1. PrimStack — which layers have specs for this PRIM ─────────────
for spec in prim.GetPrimStack():
    print(f"Layer: {spec.layer.identifier}")
    print(f"Specifier: {spec.specifier}")   # def / over / class
    print(f"TypeName:  {spec.typeName}")     # Xform, Mesh, etc.
# First spec = strongest layer contributing to this prim
# Use for: wrong type, missing prim, wrong specifier

# ── 2. PropertyStack — which layers have opinions on an ATTRIBUTE ────
attr = prim.GetAttribute("xformOp:translate")
for spec in attr.GetPropertyStack(Usd.TimeCode.Default()):
    print(f"Layer: {spec.layer.identifier}")
    print(f"Value: {spec.default}")
# First spec = strongest opinion = the winning value
# Use for: wrong property value

# ── 3. Usd.Debug.PrintComposition — most detailed arc info ───────────
Usd.Debug.PrintComposition(prim)
# Prints full composition arc breakdown for the prim
# Shows which layers contribute and how they combine
# The MOST detailed programmatic composition debugging tool

# ── 4. HasAuthoredValue — schema fallback vs explicit Set() ─────────
print(attr.HasAuthoredValue())
# True  = someone explicitly Set() this — it's intentional
# False = value is the schema fallback — nobody has touched this

# ── 5. GetPrimIndex — full composition graph ─────────────────────────
index = prim.GetPrimIndex()
print(index.DumpToString())
# Dumps complete composition graph as text
# Very verbose but comprehensive for hard-to-trace bugs

# ── EXAM TRAP: GetPrimAtPath does NOT ignore composition ─────────────
composed_prim = stage.GetPrimAtPath("/World/Chair")
# Returns the FULLY COMPOSED prim — all arcs already applied
# It does NOT isolate individual arc contributions
# Use it to: check if a prim exists, inspect composed values
# Do NOT use it to: isolate composition issues (wrong tool for that)
GetPrimAtPath is NOT for isolating composition — exam trap

Q46 specifically tested this. "UsdStage::GetPrimAtPath() retrieves prims ignoring composition arcs" is WRONG. GetPrimAtPath returns the fully COMPOSED prim — all composition arcs already applied. It is useful for verifying a prim exists and reading composed values. It is NOT useful for isolating which individual arc is contributing what — use PropertyStack or the composition debugger for that.

06 — Flattening as a Debug Tool

The exam tests three different flattening APIs and distinguishes what each one does. These are NOT the same thing.

UsdUtils.FlattenLayerStack(stage)

Merges all sublayers in the root layer stack into one. Does NOT resolve references, payloads, or variants. Returns a SdfLayer. Used purely for debugging sublayer conflicts.

stage.Flatten()

Resolves ALL composition arcs — references, payloads, variants, sublayers. Returns a complete single-layer representation. Shows the fully composed scene. Used to spot unexpected composition results.

usdcat --flatten

CLI equivalent of stage.Flatten(). Writes the flattened result to stdout or a file. Useful for quick inspection without writing Python. Same full composition resolution as stage.Flatten().

usdcat --print-composition

CLI equivalent of Usd.Debug.PrintComposition. Prints the composition arc graph for each prim. Does NOT flatten — it shows the ARCS not the resolved values. Used to understand arc structure from command line.

Python + CLIAll four flattening and composition inspection approaches
from pxr import Usd, UsdUtils

stage = Usd.Stage.Open("scene.usda")
# scene.usda has sublayers: [fx.usda, anim.usda, layout.usda, model.usda]
# And references: chair.usda at /World/Chair

# ── FlattenLayerStack — sublayers only ──────────────────────────────
flat_layer = UsdUtils.FlattenLayerStack(stage)
# flat_layer has all four sublayers merged
# BUT /World/Chair still shows the reference arc
# References/variants/payloads are NOT resolved
print(flat_layer.ExportToString())

# ── stage.Flatten() — full composition ───────────────────────────────
flat_stage_layer = stage.Flatten()
# Everything resolved: sublayers + references + variants + payloads
# /World/Chair now shows actual geometry, not a reference arc
# Use to check what the FINAL fully composed scene looks like
print(flat_stage_layer.ExportToString())

# ── CLI equivalents ──────────────────────────────────────────────────
# usdcat --flatten scene.usda                → full flatten to stdout
# usdcat --flatten scene.usda -o flat.usda   → full flatten to file
# usdcat --print-composition scene.usda      → show arc graph

# ── WHEN TO USE WHICH ────────────────────────────────────────────────
# Wrong sublayer value?     → FlattenLayerStack to merge sublayers
# Unexpected composed value? → stage.Flatten() / usdcat --flatten
# Don't understand arc structure? → PrintComposition / --print-composition
# Need to deliver an asset? → stage.Export() (not a debug tool)
Flattening for debugging vs flattening for delivery — different things

In Day 4 notes, flatten was covered as a DELIVERY tool — flatten to remove composition for a vendor. Here it is a DEBUGGING tool — flatten to see what the composed scene actually looks like. Same operation, different purpose. The exam tests both contexts. As a debug tool, flattening reveals unexpected composition results by showing the final resolved scene. As a delivery tool, it removes the ability to switch variants or update references.

07 — Namespace + Path Collision Debugging

Namespace conflicts and path mismatches are one of the most commonly tested causes of unexpected composition. The exam tests whether you know what they cause and how to detect them.

Real-world analogy

A namespace collision is like two employees at a company having identical names — John Smith in Finance and John Smith in Engineering. When a message is sent to "John Smith," the system doesn't know which one to route it to. In USD, two prims with the same path in overlapping layers will compose their opinions together — if you didn't intend that, you get unexpected overrides.

What namespace collisions cause

Collision symptoms

  • Prims appear at unexpected locations
  • Properties have wrong values due to unintended override
  • Prims missing entirely when they should exist
  • Geometry from one asset "bleeding into" another
  • References not resolving to the expected prim

How to detect and fix

  • Check all referenced files use unique prim paths
  • Use usdview Scenegraph to look for unexpected prims
  • Run GetPrimStack() to see all layers contributing
  • Use namespacing conventions (/World/Geometry/, /World/Lights/)
  • Ensure referenced assets have their defaultPrim set correctly
PythonDetecting namespace and path collision issues
# ── SCENARIO: Two references both define /World/Chair ────────────────
#
# scene.usda references both:
#   office_chair.usda  → has /World/Chair (Xform, wooden legs)
#   dining_chair.usda  → also has /World/Chair (Xform, metal legs)
#
# Both land at /World/Chair in your scene → COLLISION
# Opinions from both compose together unexpectedly
# Result: wrong chair, missing properties, or corrupt geometry

# ── HOW TO DIAGNOSE ──────────────────────────────────────────────────
prim = stage.GetPrimAtPath("/World/Chair")

# PrimStack shows ALL layers contributing — if two asset files show up
# when you expected only one, you have a path collision
for spec in prim.GetPrimStack():
    print(f"Contributing layer: {spec.layer.identifier}")
# If you see both office_chair.usda AND dining_chair.usda → collision found

# ── FIX ──────────────────────────────────────────────────────────────
# Reference them at different paths:
# /World/OfficeSide/Chair  ← from office_chair.usda
# /World/DiningSide/Chair  ← from dining_chair.usda
# Now they occupy different namespace positions — no collision

# ── PATH MISMATCHES ──────────────────────────────────────────────────
# Trying to override /World/chair (lowercase c)
# But asset defines /World/Chair (uppercase C)
# USD is case-sensitive — these are completely different paths
# Result: override never applies, original value persists
# Fix: match case exactly — /World/Chair not /World/chair

08 — Schema Version Validation

This topic was in Q47 but was completely missing from the original Day 9 notes. Schema version mismatches across layers can cause subtle, hard-to-diagnose composition errors.

Real-world analogy

Imagine a contract written using an old legal standard, and amendments written using a new legal standard where some clause definitions changed. The words are the same but the meaning has shifted. When you combine the original contract with the amendments, the result is unexpected because the two documents don't agree on what certain terms mean. Schema version mismatches work exactly the same way.

What schema version mismatches cause

When layers in the same stage were authored with different versions of a USD schema, attribute definitions may differ — different default values, renamed attributes, or changed relationships. When these layers compose together, the result can be unexpected because the opinions don't agree on the structure of the data.

Python + ConceptChecking schema versions across layers
# ── WHY SCHEMA VERSION MISMATCHES CAUSE COMPOSITION ISSUES ───────────
#
# SCENARIO:
# Layer A (old schema v1.0):
#   attribute "shader:roughness" = 0.5
#
# Layer B (new schema v2.0 where attribute was renamed):
#   attribute "inputs:roughness" = 0.3
#
# These are treated as DIFFERENT attributes — no override occurs
# The material appears to have two roughness values
# Or the wrong one is used depending on which schema version renders
#
# ── HOW TO VALIDATE ──────────────────────────────────────────────────
from pxr import Usd

stage = Usd.Stage.Open("scene.usda")

# Check what schema version each layer was authored with
for layer in stage.GetLayerStack():
    metadata = layer.GetPseudoRoot()
    # Look for customData or schema version metadata in each layer
    print(f"Layer: {layer.identifier}")

# Run usdchecker — it reports schema version incompatibilities
# usdchecker scene.usda
# [WARNING] Layer uses schema version X but current version is Y

# ── EXAM KEY POINT ────────────────────────────────────────────────────
# "Validate the USD schema versions across all layers"
# is a CORRECT strategy for unexpected composition behaviour
# Schema version mismatches cause attribute naming and structure

09 — Layer Offsets + Time-Based Composition

Q53 tested that layer offsets are NOT ignorable. This was missing from the original notes.

Exam trap — Q53

"Ignoring the layer offsets as they do not affect composition results" is INCORRECT. Layer offsets apply time transformations to referenced content. They DO affect composition — specifically for time-varying data (animation, simulation). A layer offset of offset=100, scale=2.0 shifts and scales all time samples from that layer. Ignoring offsets means you will not understand why animated data appears at the wrong time.

ConceptHow layer offsets affect time-based composition
# ── LAYER OFFSETS AFFECT TIME-VARYING DATA ───────────────────────────
#
# A reference with a layer offset:
# prepend references = @./anim.usda@ (offset=100, scale=2.0)
#
# An animation keyframe at time=10 in anim.usda
# appears at time = (10 × 2.0) + 100 = 120 in the composed stage
#
# If you ignore layer offsets during debugging:
#   You think the keyframe should be at time=10
#   But you see it at time=120
#   Mystery bug → actually just the layer offset
#
# ── HOW TO CHECK LAYER OFFSETS ──────────────────────────────────────

stage = Usd.Stage.Open("scene.usda")
prim  = stage.GetPrimAtPath("/World/AnimChar")

for spec in prim.GetPrimStack():

Complete exam trap answers — things that are always WRONG

Exam elimination guideEliminate these immediately when you see them
# ALWAYS WRONG — eliminate immediately:
#
# "GetPrimAtPath retrieves prims ignoring composition arcs"
# → WRONG — GetPrimAtPath returns the FULLY COMPOSED prim
#
# "Directly editing the composed stage's in-memory representation"
# → WRONG — composed stage is READ-ONLY. Edit underlying layers.
#
# "Clear the USD cache or restart runtime to reset composition"
# → WRONG — USD composition is deterministic. Cache = performance only.
#
# "Manually edit root layer to remove all references and start over"
# → WRONG — destructive. Does not address root cause systematically.
#
# "Delete all payloads and reload the stage"
# → WRONG — destructive. Deletes data. Does not fix composition conflicts.
#
# "Modify the USD schema to force specific opinions"
# → WRONG — schemas define data structure, not composition behaviour.
#            Modifying schemas leads to instability and incompatibility.
#
# "Ignore layer offsets as they do not affect composition"
# → WRONG — layer offsets affect time-varying data composition.
#
# "usdchecker automatically fixes composition issues"
# → WRONG — usdchecker VALIDATES, it does NOT auto-fix composition.
#
# "Rely solely on GetSessionLayer() for all composition debugging"
# → WRONG — session layer only contains ephemeral overrides, not the
#            full composition. Relying only on it misses other layers.
#
# "Flatten the entire stage to minimize layer composition overhead"
# (as a PERFORMANCE OPTIMISATION, not a debug tool)
# → WRONG — flattening INCREASES memory and removes layering benefits.
#            Use Hydra render delegates for performance optimisation.
